Loki 入门

发表于 2021-08-18 更新于 2021-11-24 分类于 运维 阅读次数： 本文字数： 5.5k 阅读时长 ≈ 5 分钟

Loki 是一个功能齐全的日志聚合系统的核心, 专为高效保存日志消息而优化, 与其它日志系统不同, Loki 不索引原始日志消息, 而是基于标签构建索引

Loki 的组件在内部称为模块 (module), 单体模式部署时, 每个组件都运行在同一进程中, 通过 Loki 配置的 gRPC 端口在本地网络中连接, 以进行组件之间的相互通信, Loki 主要由 5 个组件构成:

  • Distributor 负责处理客户端传入的流, 这是日志数据写入路径中的第一站, 分发器将验证每个流的正确性, 确保其在配置的租户 (或全局) 限制内, 然后将有效的数据块分批, 并行发送到匹配的 Ingester
  • Ingester 负责接收数据块, 当数据达到一定数量或期限时, 将数据块刷新到数据库中, 其对数据块和索引分别使用单独的数据库, 因为它们的数据类型不同
  • Querier 基于 LogQL 查询语言, 以检索存储库和 Ingester 中尚未刷新的数据的索引, 并返回匹配的日志查询结果
  • Query Frontend 负责提供查询的 API 端点, 传入的查询请求保存在内部的队列中, 以便 Querier 提取执行
  • Ruler 分为记录规则和告警规则, 用于预计算表达式和触发告警事件

Loki 提供了多种安装方式, 详见 installation, 支持使用容器方式部署 grafana/loki

Loki 通过 YAML 文件来配置, 详见 configuration

Loki 不附带任何内置的身份认证层, 如需对外暴露端口的方式部署, 需要在服务前置运行身份认证的反向代理, 例如使用 basic auth 或 OAuth2 代理的 Nginx 服务

通过运行二进制文件 release 启动服务, 详见 local

Loki 提供了一组管理 API, 以促进自动化和集成, 详见 management-api

AlertManager 入门

发表于 2021-07-19 更新于 2021-07-27 分类于 运维 阅读次数： 本文字数： 5.3k 阅读时长 ≈ 5 分钟

AlertManager 负责处理客户端应用程序 (如 Prometheus 服务等) 发送的告警事件, 根据配置定义的规则, 可以对告警事件做分组, 抑制, 静默等处理, 以及触发邮件, 第三方平台 (如 PagerDuty, Opsgenie 等), 网络端口 (webhook), 微信等多种通知方式

AlertManager 的架构可分为以下 6 个模块:

  1. API 模块用于接收客户端发送的告警请求, 如 Prometheus 服务发送的基于 http 协议的告警事件
  2. Alert Provider 模块用于存储告警内容, 默认实现存储在本机内存中, 可以通过扩展其接口, 实现将告警信息持久化到数据库或云存储
  3. Dispatcher 模块通过订阅方式从 Alert Provider 模块中获取新的告警, 并根据配置文件中的规则, 将告警信息按标签路由到不同的分组中, 以实现告警事件的分组处理
  4. Notification Pipeline 模块是一个责任链模式的组件, 通过一系列逻辑 (如抑制, 静默, 去重等) 来优化告警的质量
  5. Silence Provider 模块用于提供静默处理的实现, 静默规则可用来关闭部分告警的通知, 并通过 Cluster 以日志形式广播给集群中的节点, 避免告警事件在集群中重复出现
  6. Notify Provider 模块用于提供通知的具体实现, 并会在本地记录日志

AlertManager 提供了预编译的二进制文件 releases, 也支持使用容器的方式部署 prom/alertmanager, 详见 install

AlertManager 通过 YAML 文件来配置, 命令行可以配置不可变的系统参数, 而配置文件定义的限制规则用于通知路由和接收者, 详见 configuration

AlertManager 支持 basic 认证和 TLS, 通过 --web.config.file 指定配置文件, 详见 authentication

密码参考 bcrypt

通过运行 alertmanager 执行文件启动服务

Micrometer

发表于 2021-07-15 更新于 2021-07-20 分类于 运维 阅读次数： 本文字数： 9.4k 阅读时长 ≈ 9 分钟

Micrometer 是基于 JVM 应用的度量监控库, 它提供了应用内统一的指标采集接口和实现, 允许开发人员选择不同的数据规范对接方式, 从而最大限度的提高了对接不同监控系统的可移植性

由于不同的监控系统对数据规范的特征支持不一, Micrometer 按 3 个特征对已支持的监控系统做了划分, 详见 monitoring-systems

  • 维度(dimensional) 对于支持维度的监控系统, 维度信息将以键值对的标签形式返回, 对于不支持维度的监控系统, 维度信息将会展平添加到指标名称中
  • 速率聚合(rate aggregation) 对于不支持速率聚合的监控系统, 增量采样的数据可以进行速率聚合再返回新样本, 对于支持速率聚合的监控系统, 直接返回增量数据
  • 发布(publishing) 支持不同的监控系统对采样数据的推模式和拉模式

Micrometer 为不同的监控系统提供了称为 注册表(registry) 的实现模块和测试套件, 注册表接口 MeterRegistry 定义了创建和保存应用要采集的指标元数据的方法, 每个支持的监控系统都有一个该接口的具体实现, 详见 registry

Micrometer 默认提供 SimpleMeterRegistry 的简单实现类, 在开发者没有首选的监控系统前, 可以通过该实现类编写采样的代码, 该实现类将采样数据保存在内存中, 并且不会提供对外暴露对接数据的方式

注册表可以通过配置指标过滤器, 控制指标的注册方式和其采样数据, 详见 meter-filters

过滤器有三个基本功能:

  1. 禁用已注册的指标

  2. 转换指标的 Id 对象, 其包含指标名称和标签集

  3. 为指定类型的指标配置分布统计

Prometheus 数据采集

发表于 2021-07-04 更新于 2021-11-09 分类于 运维 阅读次数： 本文字数： 6.6k 阅读时长 ≈ 6 分钟

Prometheus 服务采集数据, 通常可以分为以下三种方式:

  • 独立方式 使用 Exporter 以零侵入的方式单独部署, 作为代理层暴露第三方系统或应用中的数据
  • 集成方式 使用 ClientLibs 耦合到应用的开发源码中, 主要用于业务相关指标的采集
  • 网关方式 使用 Pushgateway 作为中间网关, 接收应用主动推送的数据, 再暴露给服务端

Prometheus 服务拉取数据的规范基于简单的文本格式, 详见 exposition-formats

现成的 exporter 列表, 详见 exporter, 自定义编写 exporter 的示例, 参考 writing-exporters

常用系统和应用的 exporter 的配置部署示例:

  1. node-exporter 基于 Linux 内核的系统指标采集, 详见 node-exporter-guides

  2. mysqld-exporter 基于 MySQL 的数据库指标采集

  3. redis-exporter 基于 Redis 的数据库指标采集

  4. google-cadvisor 基于一个正在运行的 Docker 守护进程, 用于收集, 聚合, 处理和导出进程和容器的指标, 它为每个容器记录资源隔离参数, 资源使用情况, 网络统计信息等数据, 并支持按 Prometheus 的规范格式导出, 详见 cadvisor-prometheus

Prometheus 提供了与应用开发语言相匹配的客户端库, 详见 clientlibs, 辅助开发的工具库列表, 详见 utilities, 自定义编写客户端库的示例, 参考 writing-clientlibs

已集成 prometheus 采集规范的应用列表, 详见 software

Prometheus API

发表于 2021-07-01 更新于 2021-07-03 分类于 运维 阅读次数： 本文字数： 22k 阅读时长 ≈ 20 分钟

Prometheus 控制台的对象信息查询和表达式查询, 是通过服务端提供的基于 http 协议的 api 接口实现的, 当前稳定的接口可以在服务端的 /api/v1/ 路由端点下访问, 详见 http-api

接口响应格式基于 Json 形式, 每个成功的请求都会返回一个 2xx 的状态码, 可能返回的其它状态码如下:

  • 400 参数缺少或不正确时
  • 422 表达式不能被执行时
  • 503 服务超时或中止时

表达式查询分为瞬时和区间 2 个接口, 详见 expression-queries

  1. 瞬时查询接口, 用于在单个时间点计算表达式的结果

  2. 区间查询接口, 用于在一段时间内计算表达式的结果

当表达式查询成功时, 对应 PromQL 计算结果的 4 种类型, 响应体中 data 字段的返回值如下:

  • 区间向量(range vectors)

  • 瞬时向量(instant vectors)

  • 标量(scalars)

  • 字符串(strings)

指标元数据查询接口, 用于查询指标的类型和描述信息, 详见 querying-metric-metadata

Prometheus 数据模型

发表于 2021-06-25 更新于 2021-11-11 分类于 运维 阅读次数： 本文字数： 17k 阅读时长 ≈ 15 分钟

每种监控系统都有自己对指标的一套定义和规范, 指标的数据格式将直接影响对数据的采集和存储, 所以定义指标时需要充分考虑通用性和扩展性, Prometheus 基本上以时间序列(time series) 的形式存储所有数据, 详见 data-model

Prometheus 从监控目标的端点拉取采样的指标数据, 指标数据以文本形式组织, 每个指标都占用一行, 除了文本的最后一个空行, 其他空行都将被忽略

时间序列的组成:

  • 指标(metric)
    • 指标名称(metric name) 用于反映指标的含义, 通常描述收集的时间序列的一般性质
    • 标签集(labels) 通过键值对的形式提供多种特征维度
  • 样本(sample)
    • 时间戳(timestamp) 样本的采集时间, 精确到毫秒
    • 样本值(value) 类型为 float64 的数值, Prometheus 没有对样本值的单位进行定义, 需要使用者自己区分或者事先定义好样本数据的单位

指标名称实际上以标签的形式保存在数据库中, __name__ 是特定的标签(系统保留关键字), 用于表示指标名称, 这种方式是系统内部的表现形式, 官方推荐只能在系统内部使用

Prometheus 指标分为 Counter(计数型), Gauge(测量型), Histogram(直方图) 和 Summary(摘要型) 这 4 种类型, 详见 metric-types

Counter 计数型的特点是只增不减, 具有很好的不相关性, 不会因为机器重启而置零, 虽然它们是随着时间增加而不会减少的数字, 但有时可以将其重置为零并再次开始递增, 常见的监控指标如服务的请求数, 已完成的任务数, 错误发生的次数等

Prometheus 入门

发表于 2021-06-21 更新于 2021-10-29 分类于 运维 阅读次数： 本文字数： 11k 阅读时长 ≈ 10 分钟

Prometheus 主要用于提供近实时的, 基于动态云环境和容器的微服务, 服务和应用程序的内省监控, Prometheus 专注于现在正在发生的事情, 而不是追踪数周或数月前的数据, 它基于这样一个前提, 即大多数监控查询和警报都是从最近的数据中生成的

Prometheus 主要由 6 个核心模块构成:

  • Prometheus Server 是最核心的模块, 它主要包含抓取, 存储和查询这 3 个功能, 通过静态配置或服务发现监控的目标, 周期性的从 Prometheus Targets 或 Pushgateway 拉取监控指标数据, 数据通过一定的规则清理和数据整理后, 存储到新的时间序列中进行持久化, 之后客户端就可以通过 PromQL 语句对数据进行查询
  • Service Discovery 支持通过文件配置静态地址或服务发现的目标, 相对于直接使用文件配置, 在云环境以及容器环境下更多的监控对象都是动态的, 实际场景下, Prometheus 更适合云以及容器环境下的监控需求
  • Prometheus Targets 是 Prometheus 监控的对象, 对于长时间执行的 Job, 可以使用 Exporter 集成进行监控, Exporter 的机制是将第三方系统的监控数据按照 Prometheus 的格式暴露出来, Prometheus 是拉模式为主的监控系统
  • Pushgateway 是用于支持临时性 Job 主动推送指标的中间网关, 它使用推模式收集数据, 本质上是用于解决 Prometheus 服务器和监控目标有网络隔离的场景
  • Dashboard 除了内置的查询语言 PromQL 以外, Prometheus 还支持表达式浏览器和基于表达式浏览器的数据图形界面, 实际工作中使用 Grafana 等作为前端展示界面
  • AlertManager 是独立于 Prometheus 的一个告警组件, 需要单独安装部署, AlertManager 通过接收 Prometheus 推送过来的告警, 用于管理, 整合和分发告警到不同的目的地, AlertManager 提供了多种内置的第三方告警通知方式, 同时还提供了对 Webhook 通知的支持

Prometheus 官网提供了预编译的二进制文件 releases, 也支持使用容器的方式部署 prom/prometheus, 详见 installation

除了单独安装外, 部署 Prometheus 的最佳方式在很大程度上取决于服务器的实际环境, 开发者可以使用配置文件来构建自己的部署并将 Prometheus 打包成服务

GraphQL 接口

发表于 2021-05-20 更新于 2021-06-10 分类于 开发 ， Java 阅读次数： 本文字数： 4.8k 阅读时长 ≈ 4 分钟

使用 GraphQL 时, 最常见被选用的跨端协议是 HTTP, 通常使用单个端点来提供服务的完整功能, 服务通常使用文本格式的 Json 语法返回数据, serving-over-http 页面提供了使用 HTTP 服务的最佳实践

GraphQL 规范特意忽略了网络处理的问题, 但这并不意味着在使用 GraphQL 时没有针对这些问题的解决方案, 可代以工程上通行的准则来实现, 这些准则是基于服务端与 relay 客户端兼容来写的, 但是对于任何客户端都有用

GraphQL 服务在单个入口端点, 通常是 /graphql 上运行, 并且所有提供服务的 GraphQL 请求都应被导向此入口端点

虽然 GraphQL 也可以暴露多个资源端点, 但这可能导致开发人员在使用 GraphiQL 等工具时遇到困难

GraphQL 服务应能够处理 GET 和 POST 方法

通常建议支持另外两种情况:

  1. 如果存在 query 这个查询字符串参数, 则应当以与 GET 相同的方式进行解析和处理
  2. 如果存在 Content-Type: application/graphql 请求头, 则将 POST 请求体内容视为 GraphQL 查询字符串

无论使用任何方法发送查询和变量, 响应都应当以 Json 格式返回, 注意需按 规范 的条目规则

发表于 2021-05-17 更新于 2021-08-15 分类于 开发 阅读次数： 本文字数： 5.5k 阅读时长 ≈ 5 分钟

GraphQL 是一种用于 API 的查询语言, 也是一个满足数据查询的运行时, GraphQL 对 API 中的数据提供了一套易于理解的完整描述, 使得客户端能够准确的获得它需要的数据, 而且没有任何冗余, 也让 API 更容易的随着时间推移而演进, 规范详见 spec

GraphQL 使用模板定义语言(Schema Definitin Language, SDL) 用于定义强类型的文档, 无论使用何种语言或框架构建应用程序, GraphQL SDL 都是相同的, GraphQL Schema 定义了应用程序中的可用类型, 服务端和客户端通过它来验证 GraphQL 请求

GraphQL schema 中最基本的单元是类型, 每个 GraphQL 服务都会定义一套类型, 详见 types

初始类型(initial types) 定义了 schema 的入口, GraphQL 有 3 种初始类型, 详见 initial

  • 查询(Query)
  • 更改(Mutation)
  • 订阅(Subscription)

标量类型(scalar types) 是最基本的类型, GraphQL 有 5 种内置的标量类型, 详见 scalars

  • 整数(Int)
  • 浮点数(Float)
  • 字符串(String)
  • 布尔值(Boolean)
  • 唯一标识符(ID)

枚举类型(Enum) 是基于标量类型的变体, 其表示可能值的一个有限集, 详见 enum
