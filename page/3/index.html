BDD(Behaviour-Driven Development) 行为驱动开发是一种敏捷的软件开发技术, 使用自然语言编写用例的好处是能够让非技术人员 (客户和用户) 看懂测试用例, 可以最大程度的减少技术语言和领域语言的翻译成本

Cucumber 是一款支持行为驱动开发, 读取以纯文本形式编写的可执行规范, 验证该软件是否满足规范的内容, 并生成报告, 官网 Cucumber

Gherkin

Cucumber 运行时会从纯文本语言编写的 .feature 文件中读取测试规格说明, 解析需要测试的场景 scenario, 每个场景由一系列步骤 step 组成, 为了使 Cucumber 理解特性文件, 这些文件必须遵循 Gherkin 的基本语法规则

Gherkin 使用一组特殊关键字为可执行规范指定结构和含义, 为了让 Gherkin 能用多种语言书写, 关键字已经被翻译成多种语言, 为了提高可读性和流畅性, 有些语言可能对任何给定的关键字有多个翻译, 列表 languages

可通过文件头添加 # language: 确定要使用什么样的语言, 如果省略此属性将默认使用英语 en

自动化测试

发表于 2020-11-16 更新于 2021-04-30 分类于 测试 阅读次数： 本文字数： 1.2k 阅读时长 ≈ 1 分钟

自动化测试 是指通过工具录制或编写脚本模拟手工测试的过程, 并通过回放或运行脚本来执行测试, 从而代替人工检查被测软件或系统的功能是否符合预期要求

通过手工测试稳定的功能, 可以采用自动化测试, 机械重复的测试场景也可以采用自动化测试, 这样效率反而更高

手工测试

  1. 手工测试需要人的较强的思考能力, 通过人的逻辑思考来验证当前测试步骤是否正确;
  2. 在执行用例的过程中, 步骤更具有灵活性和跳跃性, 凭借个人的经验积累可以快速发现并准确定位问题;
  3. 如果版本迭代中有大量测试用例需要回归测试, 在人员有限的情况下, 在开展回归测试的过程中手工测试就会变得异常困难和吃力

自动化测试

  1. 执行用例的对象是测试脚本, 能通过提供的测试用例判断, 验证当前的步骤是否正确;
  2. 用例步骤之间的依赖性强, 在自动执行的过程中不需要思考, 因为这个过程是事先依靠人的思考来设计的;
  3. 对于复杂烦琐, 机械重复的测试, 绝对是测试利器, 更能让测试人员从繁重的工作中解脱出来, 从而节省大量时间;
  4. 自动化测试先期投入的工作量比较大, 而且对测试人员有一定的要求

误区一 自动化测试就是用工具录制和回放

通过录制确实可以生成对应语言的脚本, 但得到的不是有效的自动化测试脚本, 录制操作步骤是否正确以及生成的脚本是否稳定, 能否重复使用等, 都会直接影响整个测试结果

Jenkins CLI

发表于 2020-07-23 更新于 2021-05-06 分类于 运维 阅读次数： 本文字数： 2k 阅读时长 ≈ 2 分钟

虽然流水线脚本和传统的 Web 页面是大多数人使用 Jenkins 的主要接口, 但 Jenkins 同时还提供了命令行接口 (CLI) 以及应用程序接口 (REST API), 这些接口所能做的事情是有限的, 但是它们可以用于一些基本操作

Jenkins 的命令行接口可以通过两种方法访问:

  • 通过 ssh 命令;
  • 通过 jenkins-cli.jar 客户端

Jenkins 默认的 SSH 服务是被禁用的, 需要进入 Jenkins / Configure Global Security/ SSH Server 节点下, 指定固定端口或者随机端口来启用 SSH 服务, 可以进入 Jenkins / {User} / SSH Public Keys 节点下配置用户的公钥

使用命令行接口的另一个选择是 Jenkins 自带的 CLI 客户端, 这个客户端支持几种不同的协议, 包括 HTTP 和 SSH, 可以通过链接下载该文件

CLI 客户端更多详细用法见官方文档 CLI

使用 REST API 调用获取数据支持 3 种格式: XML, JSON 和 Python, 添加其中的一个限定符到 URL 末尾就会提供对应格式的数据

如果已经开启了跨站请求伪造保护功能, 要从 Jenkins 中获得一个 crumb 以便在请求中使用

Jenkins 架构

发表于 2020-07-09 更新于 2021-05-06 分类于 运维 阅读次数： 本文字数： 3.6k 阅读时长 ≈ 3 分钟

Jenkins 项目都是在 executor 上执行的, 假如单机足够强大, 让更多项目同时执行的方法就是增加 executor, 但单机的容量总会遇到上限, 而且还会有单节点问题, 解决办法就是将 Jenkins 项目分配到多台机器上执行

Jenkins 采用的是 master/agent 架构, Jenkins master 负责提供界面, 处理 HTTP 请求及管理构建环境, 构建的执行则由 Jenkins agent 负责

基于这样的架构, 只需要增加 agent 就可以轻松支持更多的项目同时执行, 这种方式称为 Jenkins agent 的 横向扩容

实现分布式构建最常用, 最基本的方式就是增加 agent, Jenkins agent 作为一个负责执行任务的程序, 它需要与 Jenkins master 建立双向连接, 连接方式有多种, 这也代表有多种增加 agent 的方式

进入 Manage Jenkins / Manage Nodes and Clouds 节点下, 点击左侧 New Node 创建 agent 节点, 填写节点相关信息, 默认启动方式 Launch method 有以下可选:

  • Launch agent agents via SSH 通过 master 以 ssh 连接方式启用代理节点;
  • Launch agent by connecting it to the master 代理节点就绪时连接 master;
  • Launch agent via execution of command on the master 通过 master 执行命令来启用代理节点;
  • Let Jenkins control this Windows slave as a Windows service 通过 master 启用 Windows 代理节点

当使用第 2 种方式创建代理节点时, 需要在代理主机上运行下载的 agent.jar, 运行成功后该节点通过 Java Web Start (JWS) 方式连接 master, 因此需要配置 Jenkins 的代理协议

进入 Manage Jenkins / Configure Global Security / Agents 节点下, 选择开放固定端口或随机开放端口来提供 TCP 通信服务, 代理协议 Agent protocols 勾选与 JWS 相关的协议

制品 是软件开发过程中产生的多种有形副产品之一, 狭义的制品可以简单地理解为二进制包, 广义的制品还包括用例, UML 图, 设计文档等

最简单的制品管理仓库就是将制品统一放在一个系统目录结构下, 但是更多的做法是使用现成的制品库, 目前现成的制品库有 Nexus, Artifactory 等

Maven 团队提供的官方插件支持将打包后的制品发布到配置的仓库中

Nexus 提供了 Nexus Platform 插件发布 Maven 制品, 安装好插件后, 在 Manage Jenkins / Configure System / Sonatype Nexus 节点下配置 Nexus 服务器地址

可以使用 archiveArtifacts 指令对制品进行归档, 构建的结果页面提供了归档文件的下载链接

在某些场景下, 需要在 pipeline 之间拷贝制品, 使用 Copy Artifact 插件可以实现

插件会将拷贝的制品的原构建序号放到环境变量 COPYARTIFACT_BUILD_NUMBER_(resultVariableSuffix) 中, 在当前 pipeline 中可以通过此变量拿到原构建序号

版本号 的制定并没有所谓的行业标准, GitHub 提出了一种具有指导意义, 统一的版本号表示规则, 称为 Semantic Versioning(语义化版本表示), 先行版本号及版本编译元数据可以加到后面作为延伸

静态代码分析 是指在不运行程序的前提下, 对源代码进行分析或检查, 范围包括代码风格, 可能出现的空指针, 代码块大小, 重复的代码等

PMD 是一款可扩展的静态代码分析器, 它不仅可以对代码风格进行检查, 还可以检查设计, 多线程, 性能等方面的问题

安装 Jenkins 的 PMD 插件可以将 PMD 报告呈现在任务详情页中

  • 指令覆盖率 (Instruction Coverage)
  • 分支覆盖率 (Branch Coverage)
  • 圈复杂度覆盖率 (Cyclomatic Complexity Coverage)
  • 行覆盖率 (Line Coverage)
  • 方法覆盖率 (Method Coverage)
  • 类覆盖率 (Class Coverage)

安装 Jenkins 的 JaCoCo 插件可以将 JaCoCo 生成的代码覆盖率呈现在任务详情页中

cURL 协议

发表于 2020-05-21 更新于 2021-05-31 分类于 工具 阅读次数： 本文字数： 7.2k 阅读时长 ≈ 7 分钟

curl 默认使用 HTTP/1.1 连接 HTTP 服务器, 当 HTTP 客户端被指定一个 URL, URL 将被分成多个部分, 并用于向服务器发起的请求中

如果路径中包含 /../ 或 /./ 这样的字符串, curl 会在将路径发送到服务器前自动压缩它们, 这些字符串的含义与本地文件系统中的路径含义是一样的

每个 HTTP 请求都可以进行身份验证, 需要身份验证的服务器会发回 401 响应码和 WWW-Authenticate: 标头, 这个标头列出了服务器支持的所有身份验证方法, 需要身份验证的 HTTP 代理会发回 407 响应码和 WWW-Authenticate: 标头, 这个标头列出了代理支持的所有身份验证方法

method

HTTP 请求都包含了一个方法, 有时也被称为动词, 具体使用哪个方法取决于使用的选项, 默认方法是 GET

选项 方法 示例
-G --get GET curl -G 'param=name' http://example.com/
-d --data
--data-binary		 POST curl -d 'param=name' http://example.com/
curl --data-binary @filename http://example.com/
-T --upload-file PUT curl -T @filename http://example.com/
-I --head HEAD curl -I http://example.com/

header

在 HTTP 请求中, 通常会有很多请求标头, 它们是一组键值对, curl 默认会在请求中添加一些标头, 具体取决于用户要求 curl 执行哪些操作

cURL 进阶

发表于 2020-05-19 更新于 2021-05-31 分类于 工具 阅读次数： 本文字数： 4.6k 阅读时长 ≈ 4 分钟

curl 很少会自己决定做什么, 它试图在最大程度上让开发人员做自己想做的事, 这是一种设计选择, 它允许通过传递选项的方式, 调整 curl 与服务器的交互

当执行 curl 命令没有返回所期望的数据时, 可以通过选项来输出更多详细的信息

使用 -v 或 --verbose 选项来启用 详细模式

当想要保存包括实际传输数据在内的完整信息时, curl 提供了两个选项

curl 提供了 --write-out 选项在传输任务完成后打印一些信息, 并且它提供了大量可添加到输出内容中的变量, 这些变量包含了与传输相关的信息

选项的可用变量见文档 NOTE

与详细模式相反的是 静默模式, 可以使用 -s 或 --silent 选项让 curl 关闭进度指示器, 而且不会在发生错误时输出任何错误信息

启用静默模式后, 可以通过使用 -S 或 --show-error 来要求它在发生错误时输出错误信息
