Jenkins 扩展插件

发表于 2020-03-30 更新于 2021-05-06 分类于 运维 阅读次数： 本文字数： 5.3k 阅读时长 ≈ 5 分钟

扩展插件

扩展

在大量使用 pipeline 后会发现 Jenkins 内置的功能并不能照顾到所有的需求, 这时就需要扩展 pipeline

函数

pipeline 本质是一个 Groovy 脚本, 所以可以在 pipeline 中定义函数, 并使用 Groovy 语言自带的特性

共享库

Jenkins pipeline 提供了 共享库(Shared library) 技术, 可以将重复代码定义在一个独立的代码控制仓库中, 其他的 Jenkins pipeline 加载使用它

创建共享库项目并推送到 SCM, 在 Manager Jenkins / Configure System / Global Pipeline Libraries 节点下配置全局共享库

共享库模板

声明式 pipeline 在 1.2 版本后, 可以在共享库中定义 pipeline, 通过此特性, 可以定义 pipeline 的模板, 如果大多数项目的结构都是标准化的, 那么利用 pipeline 模板技术可以大大降低维护 pipeline 的成本

插件

Jenkins 插件开发文档 TUTORIAL, 使用 Maven 的 archetype 插件生成代码骨架, 非常简单

Jenkins 插件开发的代码骨架中, pom.xml 文件默认添加了用于插件开发的 Maven 插件 maven-hpi-plugin, 通过该插件, 只需要实现插件代码, 然后执行 mvn hpi:run 就可以启动一个安装了插件的 Jenkins 实例, 在浏览器中打开 http://localhost:8080/jenkins 即可访问

可以配置 pipeline 按照一定的规则自动执行, 而这些规则被称为 pipeline 触发条件, Jenkins pipeline 中使用 trigger 指令来定义触发条件

需要手动触发一次任务, 让 Jenkins 加载 pipeline 后, trigger 指令才会生效

时间触发 是指定义一个时间, 时间到了就触发 pipeline 执行

定时执行 的使用场景通常是执行一些周期性的工作

需注意的是, H 应用在某一天的字段时会有不准确的情况, 因为有些月份有31天, 有些却是28天

轮询代码库 是指定期到代码仓库询问代码是否有变化, 如果有变化就执行

事件触发 是指发生了某个事件就触发 pipeline 执行

环境变量 可以看作是 pipeline 与 Jenkins 交互的媒介, 可以分为 Jenkins 内置变量 和 自定义变量

在 pipeline 执行时, Jenkins 通过一个名为 env 的全局变量, 将 Jenkins 内置环境变量暴露出来

在调试 pipeline 时, 可以在 pipeline 的开始阶段通过 sh 'printenv' 命令将 env 变量的属性值打印出来

当 pipeline 变得复杂时, 就会有自定义的环境变量的需求, 声明式 pipeline 提供了 environment 指令, 方便自定义变量

如果在 environment 中定义的变量与 env 中的变量重名, 那么被重名的变量的值会被覆盖掉

有时候, 需要定义一些全局的跨 pipeline 的自定义变量

进入 Manage Jenkins / Configure System / Global properties 节点, 添加全局环境变量, 自定义全局变量会被加入 env 属性列表中, 所以使用方法与 Jenkins 内置变量无异

参数化 是指可以通过传参来决定 pipeline 的行为

Jenkins Pipeline

发表于 2020-02-11 更新于 2021-05-06 分类于 运维 阅读次数： 本文字数： 4.3k 阅读时长 ≈ 4 分钟

Jenkins pipeline 是基于 Groovy 语言实现的一种 DSL(领域特定语言), 用于描述整条流水线是如何进行的, 流水线的内容包括执行编译, 打包, 测试, 输出测试报告等步骤

Jenkins 对这个过程进行抽象, 设计出一个基本的 pipeline 结构

基本结构满足不了现实多变的需求, 所以 Jenkins pipeline 通过各种指令来丰富自己, 指令可以被理解为对 Jenkins pipeline 基本结构的补充

Jenkins pipeline 支持的指令

指令 说明
post 在执行完成后一些附加的步骤
environment 用于设置环境变量
tools 自动下载并安装指定的工具, 并将其加入 PATH 变量
input 暂停 pipeline, 提示输入内容
options 用于配置 Jenkins pipeline 本身的选项
parallel 并行执行多个 step
parameters 执行 pipeline 前传入一些参数
triggers 用于定义执行 pipeline 的触发器
when 根据给定的条件, 决定是否执行阶段内的步骤

在使用指令时, 需要注意的是每个指令都有自己的作用域, 如果指令使用的位置不正确, Jenkins 将会报错

post 指令用于 pipeline 或 stage 完成后执行一些附加的步骤, 根据完成的状态, post 部分分成多种条件块, 并且可以同时包含多种条件块

options 指令用于配置整个 Jenkins pipeline 本身的选项

Jenkins 是一款使用 Java 语言开发的开源的自动化服务器, 通过界面或 Jenkinsfile 告诉它执行什么任务, 何时执行, 理论上, 可以让它执行任何任务, 但是通常只应用于持续集成和持续交付

Spring 007 Web 扩展

发表于 2019-07-29 更新于 2021-02-07 分类于 Spring 阅读次数： 本文字数： 7.3k 阅读时长 ≈ 7 分钟

multipart 格式的数据会将一个表单拆分为多个部分, 每个部分对应一个输入域, 在一般的表单输入域中, 它所对应的部分中会放置文本型数据, 但是如果上传文件的话, 它所对应的部分可以是二进制

DispatcherServlet 并没有实现任何解析 multipart 请求数据的功能, 它将该任务委托给了 Spring 中 MultipartResolver 策略接口的实现, 通过这个实现类来解析 multipart 请求中的内容, 从 Spring 3.1 开始, Spring 内置了两个 MultipartResolver 的实现供选择:

  • CommonsMultipartResolver 使用 Jakarta Commons FileUpload 解析请求;
  • StandardServletMultipartResolver 依赖于 Servlet 3.0 对 multipart 请求的支持 (始于 Spring 3.1)

一般来讲, 在这两者之间, StandardServletMultipartResolver 可能会是优选的方案, 它使用 Servlet 所提供的功能支持, 并不需要依赖任何其他的项目, 如果需要将应用部署到 Servlet 3.0 之前的容器中, 或者还没有使用 Spring 3.1 或更高版本, 那么可能就需要 CommonMultipartResolver 了

需要在 Servlet 中指定 multipart 的配置, 如果不设定最基本配置的话, StandardServletMultipartResolver 就无法正常工作
