cURL 协议

发表于 2020-05-21 更新于 2021-05-31 分类于 工具 阅读次数：
本文字数： 7.2k 阅读时长 ≈ 7 分钟

协议

HTTP

curl 默认使用 HTTP/1.1 连接 HTTP 服务器, 当 HTTP 客户端被指定一个 URL, URL 将被分成多个部分, 并用于向服务器发起的请求中

如果路径中包含 /../ 或 /./ 这样的字符串, curl 会在将路径发送到服务器前自动压缩它们, 这些字符串的含义与本地文件系统中的路径含义是一样的

认证

每个 HTTP 请求都可以进行身份验证, 需要身份验证的服务器会发回 401 响应码和 WWW-Authenticate: 标头, 这个标头列出了服务器支持的所有身份验证方法, 需要身份验证的 HTTP 代理会发回 407 响应码和 WWW-Authenticate: 标头, 这个标头列出了代理支持的所有身份验证方法

请求

method

HTTP 请求都包含了一个方法, 有时也被称为动词, 具体使用哪个方法取决于使用的选项, 默认方法是 GET

选项 方法 示例
-G --get GET curl -G 'param=name' http://example.com/
-d --data
--data-binary		 POST curl -d 'param=name' http://example.com/
curl --data-binary @filename http://example.com/
-T --upload-file PUT curl -T @filename http://example.com/
-I --head HEAD curl -I http://example.com/

header

在 HTTP 请求中, 通常会有很多请求标头, 它们是一组键值对, curl 默认会在请求中添加一些标头, 具体取决于用户要求 curl 执行哪些操作

cookie

HTTP cookie 是存储在客户端的键值对, 在后续请求中, 它们被发回服务器, 用于保持请求间的状态, curl 默认情况下不会使用 cookie, 需要打开 cookie 引擎才能让 curl 跟踪它接收到的 cookie, 然后在后续的请求中发送它们

响应

当 HTTP 客户端与服务器进行 HTTP 对话时, 服务器将返回 HTTP 响应消息, HTTP 响应消息包含响应码, 用来告诉客户端它是如何处理请求的一种方式

HTTP 响应消息能够以压缩的格式进行传输, 服务器通常会在响应中包含 Content-Encoding: gzip 标头, 以告诉客户端内容经过了压缩

curl 可以直接传递收到的数据, 不对其进行解码

FTP

FTP 协议是一种命令和响应协议, 客户端发送命令, 服务器做出响应, 使用 -v 选项可以看到传输过程涉及的所有命令和响应

当 FTP 客户端开始传输数据时, 它会向服务器指定要使用哪种 传输模式, curl 支持的两种传输模式是 ASCII 和 BINARY

有些 FTP 允许匿名访问, 使用 curl 进行 FTP 传输但没有指定用户名的情况下, 它将使用用户名 anonymous

目录列表

可以用 curl 列出远程 FTP 目录列表

目标文件

使用 FTP 命令遍历远程文件系统时, curl 可以通过几种不同的方式来访问目标文件

上传文件

上传文件到 FTP 服务器, 需要在 URL 中指定整个目标文件的路径和名字

curl 支持 -T 参数通配, 这样就可以上传一系列文件

SFTP

SFTP 建立在 SSH 之上的协议, 使用第三方库 libssh2 来构建 curl, 那么它就可以支持 SFTP 协议

在使用 SFTP 请求文件时, 给定的文件路径被视为远程服务器上的绝对路径, 除非特别指定要使用用户主目录的相对路径

POP3

POP3 是 Post Office Protocol 3 的简称, 即邮局协议的第 3 个版本, 它规定怎样将个人计算机连接到邮件服务器和下载电子邮件的电子协议, POP3 允许用户从服务器上把邮件存储到本地主机上, 同时删除保存在邮件服务器上的邮件

SMTP

SMTP 是简单邮件传输协议 (Simple Mail Transfer Protocol) 的简写, curl 支持将数据发送到 SMTP 服务器, 可以通过一系列命令选项将电子邮件发送给指定的一组收件人

电子邮件数据是一个文本文件, 由一组标头和正文组成, 格式见 RFC 5322

TLS

TLS 是传输层 (Transport Layer Security) 的简写, 是 TCP 之上的一个加密安全层, 基于强大的公钥加密和数字签名实现数据防篡改, 并确保服务器的可靠性

curl 支持很多协议的 TLS 版本, 如果服务器端也支持, 则可以在 curl 中使用这些协议的 TLS 版本:

  • 如果在 URL 中明确指定了协议的 TLS 版本, curl 将尝试从开始就建立 TLS 连接;
  • 如果指定了非 TLS 版本, 可以使用选项将连接升级到基于 TLS 的连接
明文 TLS 版本 使用选项
HTTP HTTPS N
LDAP LDAPS N
FTP FTPS Y
POP3 POP3S Y
IMAP IMAPS Y
SMTP SMTPS Y

CA 证书

为了确认是否与正确的 TLS 服务器通信, curl 会使用一组存储在本地的 CA 证书来验证服务器的证书签名

在 Windows 上, curl 将按以下顺序在这些目录中搜索名为 curl-ca-bundle.crt 的 CA 证书文件:

  1. 应用的目录;
  2. 当前工作目录;
  3. Windows 系统目录 (如 C:\Windows\System32);
  4. Windows 目录 (如 C:\Windows);
  5. %PATH% 中的所有目录

可以用 --cacert 选项指定要在 TLS 握手中使用的 CA 证书 (需要采用 PEM 格式) 的目录, 也可以通过环境变量 CURL_CA_BUNDLE 设置 CA 存储的完整路径

客户端证书

TLS 客户端证书是客户端以加密的方式向服务器证明自己就是真正对等方的一种方式, 在命令行中指定证书和相应的密钥, 然后在与服务器进行 TLS 握手时传递它们
