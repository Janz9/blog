Spring Web 扩展

multipart 数据

multipart 格式的数据会将一个表单拆分为多个部分, 每个部分对应一个输入域, 在一般的表单输入域中, 它所对应的部分中会放置文本型数据, 但是如果上传文件的话, 它所对应的部分可以是二进制

解析器

DispatcherServlet 并没有实现任何解析 multipart 请求数据的功能, 它将该任务委托给了 Spring 中 MultipartResolver 策略接口的实现, 通过这个实现类来解析 multipart 请求中的内容, 从 Spring 3.1 开始, Spring 内置了两个 MultipartResolver 的实现供选择:

  • CommonsMultipartResolver 使用 Jakarta Commons FileUpload 解析请求;
  • StandardServletMultipartResolver 依赖于 Servlet 3.0 对 multipart 请求的支持 (始于 Spring 3.1)

一般来讲, 在这两者之间, StandardServletMultipartResolver 可能会是优选的方案, 它使用 Servlet 所提供的功能支持, 并不需要依赖任何其他的项目, 如果需要将应用部署到 Servlet 3.0 之前的容器中, 或者还没有使用 Spring 3.1 或更高版本, 那么可能就需要 CommonMultipartResolver 了

配置

需要在 Servlet 中指定 multipart 的配置, 如果不设定最基本配置的话, StandardServletMultipartResolver 就无法正常工作

JavaConfig

如果配置 DispatcherServlet 的初始化类继承了 AbstactAnnotationConfigDispatcherServletInitia-lizer 的话, 可以通过重载 customizeRegistration 方法来配置 multipart 的具体细节

/* 重载配置 */
public class WebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer {
    @Override
    protected void customizeRegistration(ServletRegistration.Dynamic registration) {
        registration.setMultipartConfig(new MultipartConfigElement(
            "/tmp", // 上传文件的临时目录(必须配置)
            2097152, // 上传文件的最大容量(以字节为单位), 默认没有限制
            4194304, // 整个multipart请求的最大容量(以字节为单位), 默认没有限制
            0 // 上传过程中触发写入磁盘的最大容量(以字节为单位), 默认值为0
        ));
    }
}

XML

使用传统 web.xml 配置 MultipartConfigElement 的话, 可以使用 <servlet><multipart-config>

<!-- XML配置 -->
<servlet>
    <multipart-config>
        <location>/tmp</location>
        <max-file-size>2097152</max-file-size>
        <max-request-size>4194304</max-request-size>
    </multipart-config>
</servlet>

处理请求

控制器方法接收上传的文件, 最常见的方式是在某个控制器方法参数上添加 @RequestPart 注解

/* 接收multipart数据 */
@RequestMapping(value = "/form", method = RequestMethod.POST)
public String form(@RequestPart byte[] data) {
    return "home";
}

Spring 提供了 MultipartFile 接口, 它为处理 multipart 数据提供了内容更为丰富的对象

/* Multipart接口 */
public interface MultipartFile {
    String getName();
    String getOriginalFilename();
    String getContentType();
    boolean isEmpty();
    long getSize();
    byte[] getBytes() throws IOException;
    InputStream getInputStream() throws IOException;
    void transferTo(Path dest) throws IOException, IllegalStateException;
}
/* 接收MultipartFile参数 */
@RequestMapping(value = "/form2", method = RequestMethod.POST)
public String form(@RequestPart MultipartFile data) {
    return "home";
}

如果将应用部署到 Servlet 3.0 的容器中, 那么 Spring MVC 也能接受 Part 作为控制器方法的参数

/* Part接口 */
public interface Part {
    InputStream getInputStream() throws IOException;
    String getContentType();
    String getName();
    String getSubmittedFileName();
    long getSize();
    void write(String fileName) throws IOException;
}
/* 接收Part参数 */
@RequestMapping(value = "/form3", method = RequestMethod.POST)
public String form(@RequestPart Part data) {
    return "home";
}

通过 Part 参数的形式接收文件上传, 没有必要配置 MultipartResolver

测试

/* 测试 */
@Test
public void form() throws Exception {
    byte[] data = "图片数据".getBytes();
    MockMultipartFile file = new MockMultipartFile("data", "image.png", "application/x-png", data);

    mockMvc.perform(MockMvcRequestBuilders.multipart("/form").file(file))
        .andExpect(MockMvcResultMatchers.view().name("home"));

    mockMvc.perform(MockMvcRequestBuilders.multipart("/form2").file(file))
        .andExpect(MockMvcResultMatchers.view().name("home"));
}
@Test
public void form3() throws Exception {
    byte[] data = "图片数据".getBytes();
    MockPart part = new MockPart("data", "image.png", data);

    mockMvc.perform(MockMvcRequestBuilders.multipart("/form3").part(part))
        .andExpect(MockMvcResultMatchers.view().name("home"));
}

处理异常

Spring 提供了多种方式将异常转换为响应:

  • 特定的 Spring 异常将会自动映射为指定的 HTTP 状态码;
  • 异常上可以添加 @ResponseStatus 注解, 从而将其映射为某一个 HTTP 状态码;
  • 在方法上可以添加 @ExceptionHandler 注解, 使其用来处理异常

异常映射

在默认情况下, Spring 会将自身的一些异常自动转换为合适的状态码

表 Spring 的一些异常会默认映射为 HTTP 状态码

Spring 异常 HTTP 状态码
BindException 400
ConversionNotSupportedException 500
HttpMediaTypeNotAcceptableException 406
HttpMediaTypeNotSupportedException 415
HttpMessageNotReadableException 400
HttpMessageNotWritableException 500
HttpRequestMethodNotSupportedException 405
MethodArgumentNotValidException 400
MissingServletRequestParameterException 400
MissingServletRequesrPartException 400
NoSuchRequestHandlingMethodException 404
TypeMismatchException 400

对于应用所抛出的异常, Spring 提供了一种机制, 能够通过 @ResponseStatus 注解将异常映射为 HTTP 状态码

/* 自定义异常映射为HTTP状态码 */
@RequestMapping(value = "/error2", method = RequestMethod.GET)
public void error2() {
    throw new ModelNotFoundException();
}
@ResponseStatus(code = HttpStatus.NOT_FOUND, reason = "model not found")
public class ModelNotFoundException extends RuntimeException {
}
// 不使用@ResponseStatus时, 异常默认会产生500状态码(Internal Server Error)的响应

异常处理

Spring 提供 @ExceptionHandler 注解委托处理异常

/* 异常处理 */
@RequestMapping(value = "error3", method = RequestMethod.GET)
public void error3() {
    throw new ModelDuplicateException();
}
@ExceptionHandler(ModelDuplicateException.class)
public String handler(ModelDuplicateException exception) {
    System.out.println("do something ...");
}

对于 @ExceptionHandler 注解标注的方法来说, 它能够处理同一个控制器中所有处理器方法所抛出的异常

测试

/* 测试 */
@Test
public void error() throws Exception {
    mockMvc.perform(MockMvcRequestBuilders.get("/error"))
        .andExpect(MockMvcResultMatchers.status().is5xxServerError());
}

@Test
public void error2() throws Exception {
    mockMvc.perform(MockMvcRequestBuilders.get("/error2"))
        .andExpect(MockMvcResultMatchers.status().is4xxClientError());
}

public void error3() throws Exception {
    mockMvc.perform(MockMvcRequestBuilders.get("/error3"))
        .andExpect(MockMvcResultMatchers.view().name("error"));
}

控制器通知

如果控制器类的特定切面能够运用到整个应用程序的所有控制器中, 那么这将会便利很多, Spring 3.2 为这类问题引入了控制器通知 (Controller Advice)

控制器通知是任意带有 @ControllerAdvice 注解的类, 这个类会包含一个或多个如下类型的方法:

  • @ExceptionHandler
  • @InitBinder
  • @ModelAttribute

跨重定向请求传递数据

对于重定向来说, 模型并不能用来传递数据, 但是有一些其他方案:

  • 使用 URL 模板以路径变量和 / 或查询参数的形式传递数据;
  • 通过 flash 属性发送数据

URL 模板

通过路径变量和查询参数传递数据非常简单, 当构建 URL 或 SQL 查询语句的时候, 使用 String 连接是很危险的, Spring 提供了使用模板的方式来定义重定向 URL

/* URL模板 */
@RequestMapping(value = "/login", method = RequestMethod.GET)
public String login(Model model) {
    System.out.println("do something ...");
    // username作为占位符填充到URL模板中
    model.addAttribute("username", "admin");
    // userId没有匹配到占位符, 会自动以查询参数的形式附加到重定向URL上
    model.addAttribute("userId", "123456");
    return "redirect:/login/{username}";
}
@RequestMapping(value = "/login/{username}", method = RequestMethod.GET)
public String login(@PathVariable String username,
                    @RequestParam String userId) {
    return "home";
}

flash 属性

Spring 提供了将数据发送为 flash 属性的功能, 在重定向之前, 所有的 flash 属性都会复制到会话中, 在重定向后, 存在会话中的 flash 属性会被取出, 并从会话转移到模型中


@RequestMapping(value = "/login2", method = RequestMethod.GET)
public String login2(RedirectAttributes model) {
    System.out.println("do something ...");
    model.addAttribute("username", "admin");
    model.addFlashAttribute("userId", "123456");
    return "redirect:/login/{username}";
}
@RequestMapping(value = "/login2/{username}", method = RequestMethod.GET)
public String login2(@PathVariable String username, Model model) {
    if (!model.containsAttribute("userId"))
        model.addAttribute("userId", "123456");
    return "home";
}
