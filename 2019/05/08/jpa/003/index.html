JPA 基础查询

Repository

Repository 位于 Spring Data Common 的 lib 里面, 是 Spring Data 里面做数据库操作的最底层的抽象接口, 最顶级的父类, 源码里面其实什么方法都没有, 仅仅起到一个标识作用, 管理域类以及域类的 id 类型作为类型参数, 此接口主要作为标记接口捕获要使用的类型, Spring 底层做动态代理的时候发现只要是它的子类或者实现类, 都代表储存库操作

自定义方法

由于 Spring JPA Repository 的实现原理是采用动态代理的机制, 有两种定义查询方法:

  • 从方法名称中指定特定用于存储的查询和更新;
  • 通过使用 @Query 手动定义的查询

通过 @EnableJpaRepositories 的 queryLookupStrategy 属性可以配置方法的查询策略, 其中 QueryLookupStrategy.Key 的值一共有三个:

  • CREATE 直接根据方法名进行创建, 规则是根据方法名称的构造进行尝试, 一般的方法是从方法名中删除给定的一组已知前缀, 并解析该方法的其余部分, 如果方法名不符合规则, 启动的时候就会报异常;
  • USER_DECLARED_QUERY 声明方式创建, 即注解方式, 启动的时候会尝试找到一个声明的查询, 如果没有找到就将抛出一个异常, 查询可以由某处注释或其他方法声明;
  • CREATE_IF_NOT_FOUND 默认值, 以上两种方式的结合, 先用声明方式进行查找, 如果没有找到与方法相匹配的查询, 就用方法名创建规则查询

方法名规则

内部基础架构中有个根据方法名的查询生成器机制, 对于在存储库的实体上构建约束查询很有用, 该机制方法的前缀有 find...By, read...By, query...By, count...By 和 get...By, 从这些方法可以分析它的其余部分 (实体里面的字段), 引入子句可以包含其他表达式, 第一个 By 作为分隔符来指示实际标准的开始, 在一个非常基本的水平上, 可以定义实体性条件, 并与它们串联 (AND 和 OR)

属性表达式

属性表达式(Property Expressions) 只能引用托管 (泛化) 实体的直接属性, 创建及其查找的过程是, 解析算法首先将整个解释为属性, 并使用该名称检查域类的属性, 如果算法成功, 就使用该属性, 如果不是, 就拆分右侧驼峰部分的信号源到头部和尾部, 并试图找出相应的属性, 如果算法找到一个具有头部的属性, 那么它需要尾部, 并从那里继续构建树, 然后按照刚刚描述的方式将尾部分割, 如果第一个分割不匹配, 就将分割点移动到左边, 然后继续

算法可能会因为歧义选择错误的属性, 比如:

List<Person> findByAddressZipCode(String zipCode)

如果 Person 中同时有 Address 和 AddressZip, 算法可能会选择错误的属性, 要解决这个歧义, 可以在方法名称中手动定义遍历点, 比如:

List<Person> findByAddress_ZipCode(String zipCode)

Spring JPA 里面是将下划线视为保留字符, 但是强烈建议遵循标准 Java 命名约定 (不使用属性名称中的下划线, 而是使用骆驼示例)

查询结果扩展 (Projections)

/* 定义想返回字段的接口或类 */
public interface NameOnly {
    String getName();
    
    // 使用SPEL语法处理字段, 会导致select全部字段
    @Value("#{target.phone + ',' + target.email}")
    String getUserInfo();
}
// 子对象也可通过内部接口/类的形式定义
/* 返回类型使用定义的接口 */
Collection<NameOnly> findById(Collection<Long> ids);
// 可以通过泛化实现返回动态字段
Collection<T> findById(Collection<Long> ids, Class<T> type);

注解式

// Spring JPA里的优先级:
@Query > @NamedQuery > 方法名规则
// 都不支持动态条件查询

@Query

使用命名查询为实体声明查询是一种有效的方法, 对于少量查询很有效, 使用声明式 JPQL 查询有一个好处, 就是启动的时候就知道语法正确与否

/* 示例 */
@Query(value = "select u from UserDO u where u.id = ?1")
List<UserDO> findWithCustomize(Long id);
// LIKE查询
@Query(value = "select u from UserDO u where u.name like %?1%")
List<UserDO> findWithCustomize(String name);
// 原始SQL
@Query(value = "select u.* from b_user u where u.id = ?1", nativeQuery = true)
List<UserDO> findWithCustomize(Long id);
// 排序
@Query(value = "select u.name,LENGTH(u.name) as un_len from UserDO u where u.id = ?1")
List<Object[]> findWithCustomize(Long id, Sort sort);
// 分页
@Query(value = "select u from UserDO u where u.id = ?1")
Page<UserDO> findWithCustomize(Long id, Pageable pageable);
/* SpEL表达式的支持 */
@Query(value = "select e from #{#entityName} e where e.id in ?1")
List<UserDO> findById(Collection<Long> ids);
// 适合自定义公用的父类repository

@NamedQuery / @NamedNativeQuery

预定义查询

/* 示例 */
// 在Entity上定义
@NamedQuery(name = "User.findWithNamedQuery", query = "select u from UserDO u where u.id = ?1")
// 在Repository中定义
Optional<UserDO> findWithNamedQuery(Long id);
// 或者
@Query("User.findWithNamedQuery")
Optional<UserDO> findById(Long id);

@Param

默认情况下, 参数是通过顺序绑定在查询语句上的, 这使得查询方法对参数位置的重构容易出错, 为了解决这个问题, 可以使用 @Param 注解指定方法参数的具体名称, 通过绑定的参数名字做查询条件

/* 根据参数进行查询 */
@Query(value = "select u from UserDO u where u.id = :id")
Optional<UserDO> findWithParam(@Param("id") Long id);
// 不加@Param时查询参数名与方法参数名相同

@Modifying

修改查询

/* 更新指定字段 */
@Modifying(clearAutomatically = true)
@Query(value = "update UserDO u set u.phone = ?2, u.email = ?3 where u.id = ?1")
int updateWithCustomize(Long id, String phone, String email);
// Modifying的clearAutomatically属性用于设置是否刷新hibernate的一级缓存
// 自定义的update/delete语句返回为数字类型的结果, 表示执行语句后受影响的行数

@Procedure

存储过程的查询方法

/* 示例 */
// 在Entity上定义存储过程
@NamedStoredProcedureQuery(name = "UserDO.findWithProcedure", procedureName = "userFindNameById", parameters = {
        @StoredProcedureParameter(mode = ParameterMode.IN, name = "id", type = Long.class),
        @StoredProcedureParameter(mode = ParameterMode.OUT, name = "name", type = String.class)
})
// 在Repository中定义
@Procedure("UserDO.findWithProcedure")
String findWithProcedure(Long id);
// 注意, 参数名称及类型必须匹配

@QueryHints

很多数据库支持 Hint Query 的语法, Spring Data JPA 只支持一些固定的 HintValue 值, 用来优化 Query 的作用

/* 示例 */
@QueryHints(value = {@QueryHint(name = "", value = "")}, forCounting = false)
Page<UserDO> findByName(String name, Pageable pageable);
// 了解即可, 不推荐使用
