Git 分支

本文字数： 4.4k 阅读时长 ≈ 4 分钟

分支

合并

当不发生冲突时, Git 会计算合并结果, 并创建一个新的提交来代表新的统一状态, 当分支冲突时, Git 并不解决冲突, 这通常出现在对同一个文件的同一行进行修改的时候, 相反 Git 把这种争议性的修改在索引中标记为未合并(unmerged), 留给开发人员来处理, 当 Git 无法自动合并时, 需要在所有冲突都解决后做一次最终提交

退化合并

退化合并 的情况下执行合并实际都不引入新的提交, Git 只将目标分支的当前指针前移到最新的位置

当来自合并分支的所有提交都存在于目标分支上时, 即使它已经在自己的分支上前进了, 目标分支还是已经更新到最新的, 因此没有新的提交添加到目标分支上, 称为 already up-to-date

# 示例: already up-to-date
> git merge master
Already up to date.

当目标分支已经在合并分支中完全存在或表示时, 就会发生快进合并 fast-forward

# 示例: fast-forward
> git merge master
Updating 81764e6..b8e7a19
Fast-forward

常规合并

常规合并 都会产生一个最终提交, 添加到当前分支中, 表示合并的组合状态, Git 实现了 3 种不同的合并算法:

  • 解决(resolve) 解决策略只操作两个分支, 定位共同的祖先作为合并基础, 然后执行一个直接的三方合并, 通过对目标分支施加从合并基础到合并分支的变化
  • 递归(recursive) 递归策略只操作两个分支, 它能处理在两个分支之间有多个合并基础的情况, 在这种情况下, Git 会生成一个临时合并来包含所有相同的合并基础, 然后以此为基础, 通过三方合并算法导出两个给定分支的最终合并
  • 章鱼(octopus) 章鱼策略是专为合并两个以上分支而设计的, 在内部, 它多次调用 recursive 合并策略, 要合并的每个分支调一次, 然而这个策略不能处理需要用户交互解决的冲突, 在这种情况下, 必须做一系列常规合并, 一次解决一个冲突
# 示例: 指定resolve策略
> git merge -s resolve master
# 默认情况下, 不使用-s指定合并策略时, 单个合并分支使用recursive策略, 多个合并分支使用octopus策略

特殊合并

特殊合并 有两个特别的合并策略:

  • 我们的(ours) 策略合并任意数量的其他分支, 但它实际上丢弃那些分支的修改, 只使用当前分支的文件, ours 合并结果跟当前分支是相同的, 但是其他分支也记为父提交
  • 子树(subtree) 子树策略合并到另一个分支, 但是那个分支会合并到当前树的特定子树
# 示例: 指定ours策略
> git pull -s ours <repository> <refspec>

# 示例: 指定subtree策略
> git pull -s subtree <repository> <refspec>

差异

通常, 差异比较时可以通过提交名, 分支名或者标签名

# 示例: 比较暂存区和工作区
> git diff
# 一般情况下, Git 会逐行比较两个文件, 因为在源代码中, 开发人员往往修改的都是单行内容, 其邻近行通常不会被波及

# 示例: 比较暂存区和指定提交
> git diff --staged  # 默认为HEAD
> git diff --staged HEAD
# 示例: 按单词比较
> git diff --word-diff
> git diff --word-diff=color  # 以不同颜色显示

可以将提交视为自上一次提交以来项目中所纳入的一组修改, 所以版本库实质上也是一部项目的修改历史

# 示例: 比较两次提交
> git diff 01ac6b50 HEAD

# 示例: 比较提交与前次提交
> git diff 01ac6b50^!
# 示例: 指定路径限制输出版本库的子集
> git diff ./src

# 示例: 差异的统计数据, 显示文件修改的行数
> git diff --stat

排查

Git 提供了几种机制, 有助于开发人员在版本库中找到符合特定条件的提交

bisect

在开发过程中, 开发人员经常会突然遇到一个错误, 是之前早期版本在成功通过测试时没有出现的, Git 支持用二分法来搜索引发问题的提交

二分法 是基于二分搜索的一种查找方法, 查找的起点是已确认没问题的提交, 终点是已明确有错误的提交, 两者之间这段提交历史将会被分半, 位于中间的提交会在工作区中被激活, 开发人员对被激活的提交进行错误检查, 接着再根据是否在被激活提交中找到该错误的情况, 再对错误必然会隐藏的那段剩余的提交历史进行重新分半, 并检查新的中间提交, 如此反复, 最终就会找到第一次出现该错误的提交

# 示例: 执行二分法排错
> git bisect start HEAD b8e7a192
# Git使用一个分离的HEAD来管理版本库的当前检出版本, 这个分离的HEAD本质上是一个匿名分支, 第一个提交被标志成bad, 第二个提交被标志为good
# 示例: 对当前提交进行标志
> git bisect good  # 错误不在其中
> git bisect bad   # 错误就在其中
> git bisect skip  # 当前提交无法被测试
# 示例: 停止或取消二分查找
> git bisect reset
# 工作区中的内容重置回正常的开发版本

blame

Git 可以查找文件中代码的来源, 确定最终修改的开发人员以及修改的时间, 即使这些代码已被移动或复制到别处

# 示例: 查找指定文件中2-4行的提交
>  git blame -L 2,4 src\Hello.java
53d4cf4c (xxxx 2021-01-18 16:42:27 +0800 2)     public String println() {
53d4cf4c (xxxx 2021-01-18 16:42:27 +0800 3)         return "hello!"
53d4cf4c (xxxx 2021-01-18 16:42:27 +0800 4)     }
# -L 指定行范围, 可以指定多次, 允许重叠范围
# -M 检测文件中移动或复制的行
# -C 检测从同一提交中修改的其他文件移动或复制的行

pickaxe

Git 通过搜索修订版本间的差异可以找到执行改变的提交

# 示例: 查找日志中改变包含指定表达式的记录
> git log -S<string> --pickaxe-regex
> git log -G<regex>

提交

作为一般原则, 只要没有其他开发人员已经获得版本库的副本, 就可以自由的修改和完善版本库提交历史记录, 一般情况下, 如果能让提交更简洁和利于理解, 开发人员应该力求简练且结构良好的补丁, Git 提供了几个工具和命令, 专门用来修改完善版本库中的提交序列

reset

git reset 命令会把版本库和工作目录改变为已知状态

# 示例: 将HEAD指向给定提交, 默认使用--mixed模式
> git reset <commit>
# --soft   索引和工作目录的内容保持不变
# --mixed  索引改变以符合提交的树结构, 工作目录中的内容保持不变
# --hard   索引改变以符合提交的树结构, 工作目录中的内容也改变以反映提交表示的树状态

# 示例: 将HEAD指向前一次提交, 并保留工作目录中的内容
> git reset HEAD^

cherry-pick

git cherry-pick 命令会在当前分支上应用给定提交引入的变更, 这将引入一个新的提交

# 示例: 应用指定提交的变更
> git cherry-pick <commit>

# 示例: 在master分支上应用dev分支的最新提交的变更
> git checkout master
> git cherry-pick dev
# 示例: 重建一系列提交, 在master分支上倒序应用dev分支的最新3次提交的变更
> git checkout master
> git cherry-pick dev
> git cherry-pick dev^
> git cherry-pick dev~2
# 在这种情况下, 提交的顺序改变很可能不得不解决冲突, 这完全取决于提交之间的关系, 如果它们是高度耦合的, 并且修改的行有重叠, 那就有冲突需要解决

# 示例: 发生冲突时可以执行的命令
> git cherry-out (--continue | --skip | --abort | --quit)

revert

git revert 命令会在当前分支上应用给定提交的逆过程, 这将引入一个新的提交

# 示例: 撤销指定提交的变更
> git revert <commit>

# 示例: 撤销前第3次提交的变更
> git revert HEAD~3

rebase

git rebase 命令用来改变分支的基础提交

# 示例: 变基
> git rebase master

# 通常不会直接调用rebase命令, 而是对远程版本库中的修改进行变基处理
> git pull --rebase
# 示例: 将dev分支的基础提交改成master分支的最新提交
> git checkout dev
> git rebase master
# 变基操作后, 按旧基础的提交还会保留在版本库中, 虽然它们不再直接可见, 但是依然可以用过提交散列值进行访问, 只有在执行gc命令清理后才会真正从版本库中删除

# 示例: 发生冲突时可以执行的命令
> git rebase (--continue | --skip | --abort | --quit | --edit-todo | --show-current-patch)

忽略

可以通过往项目根目录下的 .gitignore 文件中添加条目的方式让 Git 忽略这些文件, 并且可以使用通配符, 也可以在项目的子目录中创建一个 .gitignore 文件, 这样该文件只能影响该目录下的文件和路径

# 示例: .gitignore
target/
pom.xml.tag
pom.xml.releaseBackup
pom.xml.versionsBackup
pom.xml.next
release.properties
dependency-reduced-pom.xml
buildNumber.properties
.mvn/timing.properties
# https://github.com/takari/maven-wrapper#usage-without-binary-jar
.mvn/wrapper/maven-wrapper.jar

模板参考 gitignore

# 示例: 忽略已经被版本化的文件
> git update-index --assume-unchanged <file>
