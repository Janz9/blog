Git 入门

发表于 2021-01-27 更新于 2021-06-06 分类于 运维 阅读次数： 本文字数： 4.7k 阅读时长 ≈ 4 分钟

入门

集中式版本控制系统 中央服务器存储着版本库的当前版本和历史记录, 每个开发者都从该服务器检出到本地工作目录, 修改后定期将修改提交到中央服务器

分布式版本控制系统 开发者环境和服务器环境之间是没有分隔的, 每个开发者同时拥有一个用于当前文件操作的工作区与一个存储该项目所以版本, 分支以及标签的本地版本库

Git

Git 是一个免费的开源分布式版本控制系统, 主要由两个层面构成, 其顶层结构就是开发者常用的命令, 称为 瓷质命令(porcelain command), 其底层结构是一组带有少量选项的简单命令, 称为 管道(plumbing)

Git 会保存每个文件原有的访问权限, 但不会保留文件的修改时间, 因此在执行检出操作时, 文件的修改时间会被设置为当前时间

许多构建工具的重新生成项目动作都是靠文件的修改时间来触发的, 由于 Git 在进行检出操作时总是用当前时间来充当文件的修改时间, 所以就能确保这些工具正确的完成整个构建过程

command

配置

Git 是一个高可配置软件, 配置文件是简单的 .ini 风格的文本文件

Git 支持不同层次的配置文件, 按照优先级递减的顺序:

  • .git/config 版本库特定的配置设置, 可用 --file 选项修改;
  • ~/.gitconfig 用户特定的配置设置, 可用 --global 选项修改;
  • /etc/gitconfig 系统范围的配置设置, 可用 --system 选项修改

更多配置参数见 config

工作区

对于一个带版本库的项目目录, 通常称之为 工作区(working trees), 工作区中所发生的修改添加到 暂存区(index), 再提交到版本库中

更多命令用法见 command

版本库

Git 版本库(repository) 是一个简单的数据库, 其中包含所有用来维护与管理项目的修订版本和历史的信息

plumbing

在版本库中, Git 维护两个主要的数据结构:

对象库(object store) 包含原始数据文件和日志消息, 以及其他用来重建项目任意版本或分支的信息

索引(index) 动态的二进制文件, 描述整个版本库的目录结构

对象库

Git 放在对象库里的对象只有 4 种类型, 这 4 种原子对象构成高层数据结构的基础:

  • 块(blob)
  • 目录树(tree)
  • 提交(commit)
  • 标签(tag)

索引

索引会记录和保存暂存变更直到准备好提交, 变更通常是添加, 删除或者编辑文件, 索引使得开发的推进与提交的变更之间能够分离开来

散列值

Git 对象库被组织及实现成一个内容寻址的存储系统, 对象库中的每个对象都有一个唯一的编码, 称之为 提交散列值(commit hash)

可以在指定某个提交散列值时采用缩写形式, 大致上只要指定几个足以识别该提交的字符就可以了, 但如果指定的字符太少, Git 还是会报错的

为了节省内存空间, Git 对于相同数据将只存储一次

Git 保存对象时使用松散(loose)的格式, 但是为了节省空间并提高效率, Git 会在以下几种情况下自动进行垃圾回收, 将多个对象打包到一个二进制文件(packfile)中

  • 版本库里有过多的松散对象;
  • 当推送(push)到一个远程版本库时;
  • 在执行一些引入许多松散对象的命令之后;
  • 在执行一些明确要求的命令(例如 git reflog expire)之后
