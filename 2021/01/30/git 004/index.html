Git 工具

发表于 2021-01-30 更新于 2021-07-13 分类于 运维 阅读次数：

工具

服务器

Git 版本库分为 裸(bare)版本库 和 开发(development, nonbare)版本库

# 示例: 版本库
> cd /d
> git init --bare bare.git              # 初始化一个裸版本库
> git clone file:///d/bare.git dev.git  # 从裸版本库克隆一个开发版本库

# 如果没有使用--initial-branch指定初始分支名称时, 则默认使用名称master

默认情况下, 每个新的克隆版本库都通过一个称为 origin 的远程版本库, 建立一个链接指回它的父版本库, 但是, 原始版本库并不知道任何克隆版本库, 也不维护指向克隆版本库的链接, 这是一个单向关系

# 示例: 查看远程版本库列表
> cd dev.git
> git remote -v
origin  file:///d/bare.git (fetch)
origin  file:///d/bare.git (push)

# origin这个名称并没有什么特别之处, 如果不想使用它, 只须在克隆操作过程中通过--origin选项指定替代名
> git clone -o <name> <repository> <directory>
> git clone --origin <name> <repository> <directory>
# 示例: 修改链接远程版本库的名称
> git remote rename <old> <new>

使用文件协议的共享方式的优点在如今许多企业环境都已经部署有现成的网络共享设备, 以作为其文件共享系统的一部分, 这对于部署一个中央版本库来说, 无疑是方便的选择, 而缺点则在于当开发人员的工作地点与中央版本库不在同一位置上时, 这个选项就很难设置了, 而且在 Git 中, 文件访问并不是最有效的一种方式

协议

daemon

标准的 Git 安装包中都会内置一个服务器程序, 该程序提供了一种简单的网络协议来访问版本库

# 示例: daemon
> git daemon --base-path=/d --enable=receive-pack
# --base-path=<path> 指定基本路径, 使URL不依赖于特定的目录
# --enable=receive-pack 默认导出的版本库只有读权限, 使用此选项打开写权限

# 示例: 在基于服务器的项目中配置
> cd bare.git
> touch git-daemon-export-ok  # 该文件允许Git提供无需授权的项目访问服务
# 示例: 克隆版本库
> git clone git://127.0.0.1/bare.git dev.git
> cd dev.git
> git ls-remote  # 显示远程版本库的引用列表
From git://127.0.0.1/bare.git
e9fa0ffc1cb77bf75b1f224d5f1e22a3f2739576        refs/heads/master

详情参考 daemon

Git deamon 提供了有效快速的从中央版本库中传输数据的方式, 而其缺点则是缺少用户验证的功能, 除此之外, 由于需要设置共享端口, 所以需要考虑防火墙的问题

http

标准的 Git 安装包中自带一个 CGI 脚本, 该脚本允许经由 Web 服务器来访问版本库

详情参考 http

在 Web 环境中访问版本库是非常方便的, 如果还想让它更安全一点, 可以使用 https 协议, 另外, 用户认证也可以通过 Web 服务器来完成, 而其缺点在于, 需要一个 Web 服务器, 并且要经由它来进行操作和管理

ssh

为了能使用 ssh 来共享版本库, 必须至少要有一台安装了 ssh 服务的计算机, 所有项目参与者要有这台服务器上的 ssh 账户, 将包含项目文件的裸版本库复制到主机上

# 示例: 配置服务器
> adduser git  # 添加用户

# 示例: 复制版本库到主机上
> scp bare.git git@127.0.0.1:/shared/bare.git
# 示例: 克隆版本库
> git clone ssh://git@127.0.0.1:/shared/bare.git dev.git

详情参考 openssh ssh

通过这种方法, 可以快速搭建一个具有读写权限, 面向多个开发人员的 Git 服务器, 需要注意的是, 目前所有开发人员都能以系统用户 git 的身份登录服务器, 需要额外修改配置加以限制

GitLab

GitLab 是一个数据库支持的 Web 应用, 相比于其他 Git 服务器, 它的安装过程涉及到更多的东西, GitLab 作为一个成熟的系统, 它提供了许多其他产品帮助协同工作, 服务器设置和运行以后, 绝大多数的管理和日常使用都可以在浏览器界面中完成

# 示例: install
> docker run --detach \
	--hostname 127.0.0.1 \
	--publish 443:443 --publish 80:80 --publish 22:22 \
	--name gitlab \
	--restart always \
	--volume $GITLAB_HOME/config:/mnt/d/gitlab/config \
	--volume $GITLAB_HOME/logs:/mnt/d/gitlab/logs \
	--volume $GITLAB_HOME/data:/mnt/d/gitlab/data \
	gitlab/gitlab-ce:latest

详情参考 gitlab configure

如果不设立自己的 Git 服务器, 可以选择将 Git 项目托管到外部专业的托管网站, 参考列表 git hosting

文件包

补丁

在有些情况下, 无法使用 Git 协议在版本库间交换数据时, Git 提供了以 补丁(patch) 的形式, 通常通过 email 来进行数据交换

# 示例: 生成补丁
> git format-patch <until>..<since>  # 指定提交范围, 不包含范围起点
> git format-patch -<n>  # 为最近n次提交生成补丁
# 默认情况下, Git为每个提交生成单独的补丁文件, 用一序列数字加上提交的日志消息为其命名

# 示例: 生成最近一次提交的补丁
> git format-path -1
0001-fix_update.patch

格式化的补丁一般是要通过电子邮件发送的, 可以直接使用电子邮件用户代理(Mail User Agent, MUA), 或者执行 Git 命令

# 示例: 配置SMTP, 以163为例
> git config --global sendemail.smtpServer smtp.163.com
> git config --global sendemail.smtpEncryption ssl
> git config --global sendemail.smtpServerPort 465
> git config --golbal sendemail.smtpPass xxxx

有许多选项来解决 SMTP 功能或问题, 将特殊的 SMTP 信息记录在全局配置文件中是很方便的, 详情参考 sendemail

# 示例: 发送补丁
> git send-email xxxx@xx.com 0001-fix_update.patch

其他开发人员通过邮件接收补丁, 储存在磁盘之后, 可以通过 Git 命令应用这些补丁

# 示例: 应用补丁
> git am 0001-fix.patch
> git am 0*.patch  # 使用通配符指定多个补丁

# 示例: 发生冲突时可以执行的命令
> git am (--continue | --skip | --abort | --quit | --show-current-patch[=(diff|raw)])

详情参考 patch

打包

在有些情况下, 如果不希望通过补丁的方式传输数量过多的提交时, Git 提供了以 打包(bundle) 的形式, 将 git push 命令所传输的所有内容打包成一个二进制文件, 可以通过邮件或闪存介质传给其他人

# 示例: 打包
> git bundle create <file> <git-rev-list-args>
# 生成的文件包含了所有重建该版本库分支所需的数据

# 示例: 打包master分支
> git bundle create bare.bundle HEAD master
# 示例: 从包中克隆版本库
> git clone bare.bundle dev

详情参考 bundle

归档

Git 提供了归档命令将项目的内容导出, 创建一个不带 Git 元数据(即不包含 .git 目录) 的压缩包

# 示例: 归档
> git archive <tree-ish> --format=<fmt> -o <file>

# 示例: 归档zip文件
> git archive HEAD --format=zip -o release.zip
> git archive HEAD target -o target.tar  # 归档指定子目录

详情参考 archive

扩展

环境变量

像通常的程序一样, Git 的常规行为依赖于环境变量

# 环境变量: 全局行为
GIT_EXEC_PATH  # Git子程序的位置
GIT_PAGER      # 控制在命令行上显示多页输出的程序
GIT_EDITOR     # Git编辑器
# 环境变量: 版本库
GIT_DIR                  # .git目录的位置
GIT_CEILING_DIRECTORIES  # 控制查找.git目录的行为
GIT_WORK_TREE            # 非空版本库的工作目录的根目录
GIT_INDEX_FILE           # 索引文件的路径
GIT_OBJECT_DEIRECTORY    # .git/objects目录的位置
# 环境变量: 路径规划
GIT_GLOB_PATHSPECS     # 控制通配符在路径规划中的默认行为, 使用通配符模式
GIT_NOGLOB_PATHSPECS   # 控制通配符在路径规划中的默认行为, 不使用通配符模式
GIT_LITERAL_PATHSPECS  # 禁用上面的两种行为
GIT_ICASE_PATHSPECS    # 让所有的路径规范忽略大小写
# 环境变量: 提交
GIT_AUTHOR_NAME      # author字段的可读名字
GIT_AUTHOR_EMAIL     # author字段的邮件
GIT_AUTHOR_DATE      # author字段的时间戳
GIT_COMMITTER_NAME   # committer字段的可读名字
GIT_COMMITTER_EMAIL  # committer字段的邮件
GIT_COMMITTER_DATE   # committer字段的时间戳
# 环境变量: 网络
GIT_CURL_VERBOSE          # Git使用curl库时显示详细信息
GIT_SSL_NO_VERIFY         # 禁用验证SSL证书
GIT_HTTP_LOW_SPEED_LIMIT  # 最低网速
GIT_HTTP_LOW_SPEED_TIME   # 最低时间, 当操作在最低网速持续最低时间, Git会终止该操作
GIT_HTTP_USER_AGENT       # 设置Git在请求HTTP时使用的user-agent请求头
# 环境变量: 比较和合并
GIT_DIFF_OPTS          # 控制git diff命令中显示的内容行数
GIT_EXTERNAL_DIFF      # 执行git diff时调用的程序
GIT_MERGE_VERBOSITY    # 控制递归合并策略的输出
# 环境变量: 调试
GIT_TRACE              # 控制常规跟踪
GIT_TRACE_PACK_ACCESS  # 控制访问打包文件的跟踪信息
GIT_TRACE_PACKET       # 打开网络操作包级别的跟踪信息
GIT_TRACE_PERFORMANCE  # 控制性能数据的日志打印
GIT_TRACE_SETUP        # 显示关于版本库和交互环境的信息
# 环境变量: 其它
GIT_SSH            # Git连接SSH时使用的程序
GIT_ASKPASS        # Git需要向用户请求验证时用到的程序
GIT_NAMESPACE      # 控制有命令空间的引用的访问
GIT_FLUSH          # 强制Git在向标准输出增量写入时使用没有缓存的I/O
GIT_REFLOG_ACTION  # 指定描述性的文字写到reflog中

详情参考 environment variables

钩子

Git 提供 钩子(hook) 可以在特定的重要动作发生时触发自定义脚本, 钩子只属于并作用于一个特定的版本库, 在克隆操作的时候不会复制, 钩子被存储在 .git/hooks 目录中, 当使用 git init 初始化一个新版本库时, Git 默认会在这个目录中放置一些示例脚本, 这些示例的名字都是以 .sample 结尾, 如果想启用它们, 得先移除这个后缀, 钩子分为以下两类:

客户端 客户端钩子由诸如提交和合并这样的操作所调用;

服务器端 服务器端钩子作用于诸如接收被推送的提交这样的联网操作

可用的钩子列表 hooks

# 文件: .git/hooks/pre-commit
#!/bin/sh
idea.bat format -r src  # 使用idea格式化代码
git add .
# 示例: 提交
> git commit -m "fix: format"
