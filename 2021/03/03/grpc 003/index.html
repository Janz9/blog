接口

grpc-spring-boot-starter 将 gRPC 与 SpringBoot 进行整合, 该项目简化了 gRPC 服务端和客户端的设置, 提供的特性仍能复用使用 gRPC 的经验, 并且允许开发者执行任何自定义操作, 详见 文档

示例

<!-- 示例: Maven配置 -->
<dependency>
    <groupId>net.devh</groupId>
    <artifactId>grpc-server-spring-boot-starter</artifactId>
</dependency>
<dependency>
    <groupId>net.devh</groupId>
    <artifactId>grpc-client-spring-boot-starter</artifactId>
</dependency>

server

将 @GrpcService 注解添加到 ImplBase 的实现类上

/* 示例: @GrpcService */
@GrpcService
public class BizImpl extends BizGrpc.BizImplBase {
}
// 配置类参考 net.devh.boot.grpc.server.config.GrpcServerProperties

服务端配置可以复用 gRPC 的 ServerBuilder 配置方式

/* 示例: 配置 */
@Bean
public GrpcServerConfigurer configurer() {
    return serverBuilder -> {
        if (serverBuilder instanceof NettyServerBuilder) {
            ((NettyServerBuilder) serverBuilder)
                    .keepAliveTime(30, TimeUnit.SECONDS)
                    .keepAliveTimeout(5, TimeUnit.SECONDS)
                    .permitKeepAliveWithoutCalls(true);
        }
    };
}

client

将 @GrpcClient 注解添加在需要注入客户端的字段上, 支持 Channel 和各种类型的 Stub

/* 示例: @GrpcClient */
@Service
public class BizImpl {
    @GrpcClient("bizService")
    private BizGrpc.BizStub bizStub;
}
// 配置类参考 net.devh.boot.grpc.client.config.GrpcChannelProperties

客户端配置可以复用 gRPC 的 ChannelBuilder 配置方式

/* 示例: 配置 */
@Bean
public GrpcChannelConfigurer keepAliveClientConfigurer() {
    return (channelBuilder, name) -> {
        if (channelBuilder instanceof NettyChannelBuilder) {
            ((NettyChannelBuilder) channelBuilder)
                    .keepAliveTime(30, TimeUnit.SECONDS)
                    .keepAliveTimeout(5, TimeUnit.SECONDS);
        }
    };
}

在注入客户端之前可以通过 StubTransformer 配置修改 Stub

/* 示例: 配置 */
@Bean
public StubTransformer call() {
    return (name, stub) -> {
        if ("inProcess".equals(name)) {
            return stub.withWaitForReady();
        } else {
            return stub;
        }
    };
}

上下文

grpc-java 中, 消息发送和请求处理的不同阶段可能在不同的线程中运行, 流式调用中也是这样, 如果需要在会话或请求中存储数据, 需要使用 gRPC 的 Context 或 grpcRequest 作用域, 详见 contextual-data

/* 示例: Context */
public static Context.Key<String> CRED_KEY = Context.key("CRED_KEY");

// A阶段
Context.current().withValue(CRED_KEY, cred);
// B阶段
String cred = CRED_KEY.get();
// 常用于拦截器中对安全认证或分布式部署时信息的存储

grpc-spring-boot-starter 项目添加了一个 grpcRequest 范围, 该功能类似 Spring Web 的 request 作用域, 它只适用于单个请求

/* 示例: grpcRequest */
@Bean
@Scope(scopeName = GrpcRequestScope.GRPC_REQUEST_SCOPE_NAME, proxyMode = ScopedProxyMode.TARGET_CLASS)
public Context.Key<String> credKey() {
    return Context.key("CRED_KEY");
}

@Autowired
private Context.Key<String> credKey;
// A阶段
Context.current().withValue(credKey, cred);
// B阶段
String cred = credKey.get();
// 常用于单个请求的数据隔离

状态码

gRPC 使用一组定义良好的状态码 statuscodes 作为 API 的一部分

/* 示例: 状态码 */
public class BizImpl extends BizGrpc.BizImplBase {
    @Override
    public void insert(Entity request, StreamObserver<Identification> responseObserver) {
        if ("".equals(request.getName())) {
            responseObserver.onError(
                // 返回参数错误的状态码
                Status.INVALID_ARGUMENT.withDescription("name is not be blank.").asRuntimeException()
            );
            return;
        }
        responseObserver.onNext(Identification.newBuilder().setId(request.getId()).build());
        responseObserver.onCompleted();
    }
}

grpc-spring-boot-starter 项目支持在切面中配置全局异常映射为 gRPC 的状态码, 详见 exception-handler

/* 示例: 全局异常 */
@GrpcAdvice
public class BizExceptionHandler {
    // 将参数错误的异常映射为状态码
    @GrpcExceptionHandler(IllegalArgumentException.class)
    public Status handler(IllegalArgumentException exception) {
        return Status.fromCode(Status.Code.INVALID_ARGUMENT)
            .withDescription(exception.getMessage())
            .withCause(exception);
    }
}

扩展

errors 谷歌设计指南中介绍了通用的错误处理, 错误模型由 google/rpc/status.proto 逻辑定义, 该实例在应用发生错误时返回给客户端, 可以在 google/rpc/error_details.proto 中找到一组标准的错误类型, 它们涵盖了对于错误处理的最常见需求

  • ErrorInfo 提供稳定又可扩展的结构化错误信息
  • RetryInfo 描述客户端何时可以重试失败的请求
  • QuotaFailure 描述配额检查失败的方式
  • BadRequest 描述客户端请求中的违规行为
/* 示例: 错误详情 */
public class BizImpl extends BizGrpc.BizImplBase {
    @Override
    public void insert(Entity request, StreamObserver<Identification> responseObserver) {
        if ("".equals(request.getName())) {
            BadRequest detail = BadRequest.newBuilder()
                .addFieldViolations(BadRequest.FieldViolation.newBuilder()
                    .setField("name")
                    .setDescription("name is not be blank.")
                    .build())
                .build()
            com.google.rpc.Status status = com.google.rpc.Status.newBuilder()
                .setCode(Status.Code.INVALID_ARGUMENT.value())
                .setMessage("validation failed.")
                .addDetails(Any.pack(detail))
                .build();
            responseObserver.onError(StatusProto.toStatusException(status));
            return;
        }
        responseObserver.onNext(Identification.newBuilder().setId(request.getId()).build());
        responseObserver.onCompleted();
    }
}

校验

参数校验功能, 参考 validator

Javax

使用 javax.validation 方案时, 由于无法向未生成的结构化类中添加约束注解, 所以需要通过 XML 配置编写消息字段的约束

<!-- 示例: Maven配置 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>

XML 配置参考 xml-configuration

<!-- 文件: META-INF/validation.xml -->
<validation-config xmlns="http://jboss.org/xml/ns/javax/validation/configuration" version="1.1">
    <constraint-mapping>META-INF/validation/biz.xml</constraint-mapping>
    <property name="hibernate.validator.fail_fast">true</property>
</validation-config>
<!-- 文件: META-INF/validation/biz.xml -->
<constraint-mappings xmlns="http://jboss.org/xml/ns/javax/validation/mapping" version="1.1">
    <bean class="lib.Entity" ignore-annotations="true">
        <getter name="id">
            <constraint annotation="javax.validation.constraints.Min">
                <element name="value">0</element>
            </constraint>
        </getter>
        <getter name="code">
            <constraint annotation="javax.validation.constraints.NotBlank"/>
        </getter>
        <getter name="name">
            <constraint annotation="org.hibernate.validator.constraints.Length">
                <element name="min">2</element>
                <element name="max">4</element>
            </constraint>
        </getter>
    </bean>
</constraint-mappings>
/* 示例: Validator */
public class BizImpl extends BizGrpc.BizImplBase {
    @Autowired
    private Validator validator;
    @Override
    public void insert(Entity request, StreamObserver<Identification> responseObserver) {
        // 参数校验
        Optional<ConstraintViolation<Entity>> opt = validator.validate(request).stream().findFirst();
        if (opt.isPresent()) {
            responseObserver.onError(Status.INVALID_ARGUMENT.withDescription(
                opt.get().getPropertyPath().toString() + opt.get().getMessage()
            ).asRuntimeException());
            return;
        }
        responseObserver.onNext(Identification.newBuilder().setId(request.getId()).build());
        responseObserver.onCompleted();
    }
}

除了 XML 配置以外, 也可以通过代码以构造的方式定义约束的映射关系, 详见 constraint-declaration

PGV

protoc-gen-validate 是一个 protoc 插件, 用于生成多语言的消息验证器, 虽然 protocol buffers 协议有效的保证了结构化数据的类型, 但它不能强制值的语义规则, 这个插件添加了生成代码的约束验证的支持

<!-- 示例: Maven配置 -->
<dependencies>
    <dependency>
        <groupId> io.envoyproxy.protoc-gen-validate</groupId>
        <artifactId>pgv-java-stub</artifactId>
    </dependency>
    <dependency>
        <groupId>io.envoyproxy.protoc-gen-validate</groupId>
        <artifactId>pgv-java-grpc</artifactId>
    </dependency>
</dependencies>
<build>
    <extensions>
        <extension>
            <groupId>kr.motd.maven</groupId>
            <artifactId>os-maven-plugin</artifactId>
        </extension>
    </extensions>
    <plugins>
        <plugin>
            <groupId>org.xolstice.maven.plugins</groupId>
            <artifactId>protobuf-maven-plugin</artifactId>
            <configuration>
                <protocArtifact>com.google.protobuf:protoc:${protoc.version}:exe:${os.detected.classifier}</protocArtifact>
            </configuration>
            <executions>
                <execution>
                    <id>protoc-java-pgv</id>
                    <goals>
                        <goal>compile-custom</goal>
                    </goals>
                    <configuration>
                        <pluginParameter>lang=java</pluginParameter>
                        <pluginId>java-pgv</pluginId>
                        <pluginArtifact>io.envoyproxy.protoc-gen-validate:protoc-gen-validate:${pgv.version}:exe:${os.detected.classifier}</pluginArtifact>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>

插件定义的可用约束的类型和规则, 详见 validate.proto

插件定义的约束在不同开发语言中的可用性, 详见 rule-comparison

"示例: proto文件"
import "validate/validate.proto";
message Entity {
  int32 id = 1 [(validate.rules).int32.gt = 0];
  string code = 2 [(validate.rules).string = {
                   pattern: "^[0-9A-Za-z]+$",
                   max_bytes: 256
                   }];
  string name = 3 [(validate.rules).string.email = true];
}
/* 示例: 拦截器方式 */
@GrpcGlobalServerInterceptor  // 服务端配置
public ServerInterceptor serverInterceptor() {
    return new ValidatingServerInterceptor(new ReflectiveValidatorIndex());
}
@GrpcGlobalClientInterceptor  // 客户端配置
public ClientInterceptor clientInterceptor() {
    return new ValidatingClientInterceptor(new ReflectiveValidatorIndex());
}
/* 示例: 代码调用方式 */
@Override
public void insert(Entity request, StreamObserver<Identification> responseObserver) {
    try {
        new ReflectiveValidatorIndex().validatorFor(request.getClass()).assertValid(request);
    } catch (ValidationException e) {
        responseObserver.onError(Status.INVALID_ARGUMENT.withDescription(e.getMessage()).asRuntimeException());
        return;
    }
    responseObserver.onNext(Identification.newBuilder().setId(request.getId()).build());
    responseObserver.onCompleted();
}

安全

grpc-spring-boot-starter 原生支持 spring-security, 详见 security

认证

服务端可以通过定义 GrpcAuthenticationReader 实现需要的身份认证, 框架提供了一些内置的实现:

  • AnonymousAuthenticationReader 匿名身份认证
  • BasicGrpcAuthenticationReader 基础身份认证
  • BearerAuthenticationReader 令牌协议的身份认证
  • SSLContextGrpcAuthenticationReader 基于证书的身份认证
  • CompositeGrpcAuthenticationReader 复合多个身份认证
/* 示例: 基础认证 */
@Bean
public GrpcAuthenticationReader authenticationReader() {
    return new BasicGrpcAuthenticationReader();
}
// 其它用户配置同spring-security

授权

服务端可以配置 GrpcSecurityMetadataSource 为每个 gRPC 的方法添加安全条件

/* 示例: 配置权限 */
@Bean
public GrpcSecurityMetadataSource security() {
    ManualGrpcSecurityMetadataSource source = new ManualGrpcSecurityMetadataSource();
    source.set(BizGrpc.getSearchMethod(), AccessPredicate.permitAll());
    source.set(BizGrpc.getInsertMethod(), AccessPredicate.authenticated());
    source.set(BizGrpc.getDeleteMethod(), AccessPredicate.hasRole("ROLE_ADMIN"));
    return source;
}
// 也可以在grpc方法级别上直接使用spring-security注解

单元测试

在单元测试中, 直接在 gRPC 服务端的 Bean 实例上调用方法, 可以使用 Mockito 等模拟库限制依赖树的深度, 从而简化测试的流程

/* 示例: 单元测试 */
@SpringBootTest
public class BizTest {
    @Autowired
    private static BizImpl bizImpl;
    @Test
    public void insert() {
        StreamRecorder<Identification> recorder = StreamRecorder.create();
        bizImpl.insert(Entity.newBuilder().setId(1).setName("").build(), recorder);
        // 断言
        Assertions.assertTrue(recorder.awaitCompletion(5, TimeUnit.SECONDS));
        Assertions.assertNull(recorder.getError(), recorder.getError().getMessage());
        Assertions.assertEquals(1, recorder.firstValue().get().getId());
    }
}

集成测试

在集成测试中, 配置启用 InProcessServer 服务, 可以在不打开端口的情况下调用 gRPC 服务, 从而以节省网络开销的方式来测试整个调用栈, 详见 in-process-server

/* 示例: 集成测试 */
@SpringBootTest(properties = {
    "grpc.server.inProcessName=test",
    "grpc.server.port=-1",
    "grpc.client.inProcess.address=in-process:test"
})
@DirtiesContext
public class BizTest {
    @GrpcClient("inProcess")
    private BizGrpc.BizBlockingStub bizStub;
    @Test
    public void insert() {
        Identification identification = bizStub.insert(Entity.newBuilder().setId(1).setName("").build());
        // 断言
        Assertions.assertNotNull(identification);
        Assertions.assertEquals(1, identification.getId());
    }
}
