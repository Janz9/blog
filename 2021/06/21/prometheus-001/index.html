Prometheus 入门

发表于 2021-06-21 更新于 2021-10-29 分类于 运维 阅读次数： 本文字数： 11k 阅读时长 ≈ 10 分钟

入门

Prometheus 主要用于提供近实时的, 基于动态云环境和容器的微服务, 服务和应用程序的内省监控, Prometheus 专注于现在正在发生的事情, 而不是追踪数周或数月前的数据, 它基于这样一个前提, 即大多数监控查询和警报都是从最近的数据中生成的

Prometheus 主要由 6 个核心模块构成:

  • Prometheus Server 是最核心的模块, 它主要包含抓取, 存储和查询这 3 个功能, 通过静态配置或服务发现监控的目标, 周期性的从 Prometheus Targets 或 Pushgateway 拉取监控指标数据, 数据通过一定的规则清理和数据整理后, 存储到新的时间序列中进行持久化, 之后客户端就可以通过 PromQL 语句对数据进行查询
  • Service Discovery 支持通过文件配置静态地址或服务发现的目标, 相对于直接使用文件配置, 在云环境以及容器环境下更多的监控对象都是动态的, 实际场景下, Prometheus 更适合云以及容器环境下的监控需求
  • Prometheus Targets 是 Prometheus 监控的对象, 对于长时间执行的 Job, 可以使用 Exporter 集成进行监控, Exporter 的机制是将第三方系统的监控数据按照 Prometheus 的格式暴露出来, Prometheus 是拉模式为主的监控系统
  • Pushgateway 是用于支持临时性 Job 主动推送指标的中间网关, 它使用推模式收集数据, 本质上是用于解决 Prometheus 服务器和监控目标有网络隔离的场景
  • Dashboard 除了内置的查询语言 PromQL 以外, Prometheus 还支持表达式浏览器和基于表达式浏览器的数据图形界面, 实际工作中使用 Grafana 等作为前端展示界面
  • AlertManager 是独立于 Prometheus 的一个告警组件, 需要单独安装部署, AlertManager 通过接收 Prometheus 推送过来的告警, 用于管理, 整合和分发告警到不同的目的地, AlertManager 提供了多种内置的第三方告警通知方式, 同时还提供了对 Webhook 通知的支持

安装

Prometheus 官网提供了预编译的二进制文件 releases, 也支持使用容器的方式部署 prom/prometheus, 详见 installation

除了单独安装外, 部署 Prometheus 的最佳方式在很大程度上取决于服务器的实际环境, 开发者可以使用配置文件来构建自己的部署并将 Prometheus 打包成服务

配置

Prometheus 通过 YAML 文件来配置, 初始的配置文件 prometheus.yml 位于执行文件的根目录, 详见 configuration

YAML 配置非常复杂, 建议通过 Prometheus 附带的代码校验工具 promtool 来验证 YAML 配置文件

可以在控制台 /config 查看当前服务使用的配置信息

认证

Prometheus 支持 basic 认证和 TLS, 通过 --web.config.file 指定配置文件, 详见 authentication

密码参考 bcrypt

使用 Nginx 反向代理 Prometheus 时, 配置 basic 认证和 TLS 的示例, 参考 basic tls

启动

通过运行 prometheus 执行文件启动服务, 详见 starting

服务启动之后, 可以在控制台 /targets 查看当前的监控目标列表

初始配置中只有一个作业 prometheus 的监控目标, 目标的监控数据从本地的 9090 端口返回指标, 可以通过配置的端点默认值 /metrics 查看抓取的指标数据

Prometheus 提供了一组管理 API, 以促进自动化和集成, 详见 management-api

  • GET /-/healthy 该端点用于显示服务的健康状态
  • GET /-/ready 当服务准备好时, 该端点返回 200 状态码
  • POST /-/reload 该端点用于触发重新加载服务的配置和规则文件
  • POST /-/quit 该端点用于触发服务正常关闭

作业

在 Prometheus 中可以获取的端点称为实例(instance), 通常对应于单个进程, 具有相同目的的实例集合称为作业(job), 详见 jobs

服务发现

在基于云的基础设施环境中, 应用可以随着定义的规则动态的创建或者销毁, 从而自适应访问规模的变化, 这种按需的资源使用方式对于监控系统而言就意味着没有一个固定的监控目标, Prometheus 的解决方案是引入服务发现(service discovery) 的模式

基于文件

基于文件的服务发现不需要依赖其它的平台或第三方服务, 因此可以通过与自动化配置管理工具结合使用, 将监控目标的信息写入文件, Prometheus 就会定时从文件中读取最新的目标信息, 详见 file-sd

基于 http

基于 http 协议的服务发现提供了更通用的方式获取目标信息, 它能够通过接口的端点获取目标信息, 详见 http-sd

基于服务中心

Prometheus 支持多种常用的服务发现组件, 如 Kubernetes, Zookeeper, Azure, Consul 等, 详见 service-discovery

规则

Prometheus 中可以定期执行的两种规则为 记录规则(recording rules) 和 告警规则(alerting rules)

可以在控制台 /rules 查看配置的规则列表, 每一条规则都提供了一个可供参考的评估时间

记录规则

记录规则可以预先计算经常需要或计算量大的表达式, 并将结果保存为一组新的时间序列, 由此实现对复杂查询语句的性能优化并提高查询效率, 查询预计算的结果通常在速率上比每次执行原始表达式要快得多, 这对于仪表板特别有用, 仪表板每次刷新时都需要重复查询相同的表达式, 详见 recording-rules

记录规则在规则组中定义, 规则组中的规则是按顺序执行的, 可以根据规则创建指标, 并在后面的规则中重用这些指标, 这个特性仅在规则组内使用, 不适用于并行关系的规则组之间

告警规则

告警规则可以基于 PromQL 表达式定义告警条件, 并将有关触发告警的通知发送到外部服务器, 只要告警表达式在给定的时间点生成一个或多个矢量元素, 告警就被这些元素的标签集视为处于活动状态, 详见 alerting-rules

在告警中可以使用模板, 模板使用标准的 Go 语法, 详见 template

可以在控制台 /alerts 查看监控中的告警列表, 告警规则有三种状态:

  • 非活动(Inactive) 没有触发阈值
  • 挂起(Pending) 已触发阈值但未满足告警的持续时间
  • 触发(Firing) 已触发阈值且满足告警的持续时间

存储

Prometheus 包括一个本地磁盘上的时间序列数据库, 但也可以选择与远程存储系统集成, 详见 storage

本地存储

Prometheus 存储层使用 Prometheus TSDB 的本地存储, 详见 prometheus-tsdb

TSDB 工具已包含在 promtool 中, 可以用于分析和监测数据库中条目的流失(churn) 和基数(cardinality)

结构

Prometheus 采集的数据默认保存在根目录的 data 文件夹中, 详见 format

  • 数据块(block)

    TSDB 中默认每 2 小时为一个时间窗口, 产生的数据存储在一个数据块中, 数据块会定期压缩, 合并和清理, 每个数据块都拥有全局唯一的名称, 这是通过 ULID 原理生成的

    • 样本数据(chunks)

      保存时序数据的目录, 按文件大小 512MB 存为多个文件, 通过数字编号命名

    • 索引(index)

      该文件是 TSDB 实现高效查询的基础, 可以通过指标名称(metrics name) 和标签(labels) 查找时序数据在样本数据中的位置

    • 元数据(meta.json)

      数据块的元数据, 该文件记录的信息用于数据块的合并和清理等操作

    • 里程碑(tombstones)

      该文件用于对数据进行软删除, TSDB 采用标记删除的策略来降低删除操作的成本

  • 预写日志(wal)

    TSDB 的关键点是先写日志再写磁盘, 使用预写日志(write-ahead logging) 技术可以方便的进行回滚, 重试数据等操作, 保证数据可靠性, 日志按文件大小 128MB 分为多个文件段, 这些文件包括还没有被压缩的原始数据, 所以比常规的数据块文件大得多

    • 检查点(checkpoint)

      该文件用于对日志数量进行控制, 它用于日志定期压缩和清理, 目的是减少宕机后恢复的时长, 降低磁盘占用率

配置

Prometheus 将时间序列及其样本存储在磁盘上, 鉴于磁盘空间是有限的资源, Prometheus 提供了几个参数来修改本地存储的配置

远程存储

Prometheus 的远程写入 remote_write 和远程读取 remote_read 功能允许透明的发送和接收样本, 这主要用于长期存储, 用户可以在配置文件中指定远程的 URL 地址, Prometheus 将采集到的样本数据通过 HTTP 的形式发送给适配器(adaptor), 而适配器则实现对接外部的远程存储系统, 外部系统可以是真正的存储系统, 公有云的存储服务, 消息队列等任意形式, 详见 remote-storage

现有集成的远程存储系统, 详见 remote-integrations

集群

单体架构并不是高可用集群, 存在发生单点故障的可能性, 而且随着监控规模的扩展, 大量数据的读写也很容易让单体架构响应变慢, 过度消耗系统资源, 因此, 在企业实际生产中, 非常有必要构建 Prometheus 的高可用集群的架构, 详见 federation

因为 Prometheus 是基于时间序列的存储及监控系统, 对系统时间的准确性要求很高, 所以在部署 Prometheus 集群之前, 必须对服务器的时间源做同步, 比如 NTP 网络授时服务

联邦集群

在每个数据中心部署单独的 Prometheus Server, 用于采集当前数据中心监控数据, 并由一个中心的 Prometheus Server 负责聚合多个数据中心的监控数据, 这一特性在 Prometheus 中称为联邦集群

联邦集群方案可以降低单个服务采集负载, 通过联邦节点汇聚核心数据, 降低单机存储压力, 避免单点故障时都体现出不错的优势

功能分区

当单个采集任务的目标数量非常庞大时, 如果简单通过联邦集群无法有效处理, 可以考虑在实例级别进行功能分区, 可以按照任务的不同实例进行划分, 以支持规模的扩展
