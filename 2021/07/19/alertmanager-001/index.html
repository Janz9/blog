AlertManager 入门

发表于 2021-07-19 更新于 2021-07-27 分类于 运维 阅读次数： 本文字数： 5.3k 阅读时长 ≈ 5 分钟

入门

AlertManager 负责处理客户端应用程序 (如 Prometheus 服务等) 发送的告警事件, 根据配置定义的规则, 可以对告警事件做分组, 抑制, 静默等处理, 以及触发邮件, 第三方平台 (如 PagerDuty, Opsgenie 等), 网络端口 (webhook), 微信等多种通知方式

AlertManager 的架构可分为以下 6 个模块:

  1. API 模块用于接收客户端发送的告警请求, 如 Prometheus 服务发送的基于 http 协议的告警事件
  2. Alert Provider 模块用于存储告警内容, 默认实现存储在本机内存中, 可以通过扩展其接口, 实现将告警信息持久化到数据库或云存储
  3. Dispatcher 模块通过订阅方式从 Alert Provider 模块中获取新的告警, 并根据配置文件中的规则, 将告警信息按标签路由到不同的分组中, 以实现告警事件的分组处理
  4. Notification Pipeline 模块是一个责任链模式的组件, 通过一系列逻辑 (如抑制, 静默, 去重等) 来优化告警的质量
  5. Silence Provider 模块用于提供静默处理的实现, 静默规则可用来关闭部分告警的通知, 并通过 Cluster 以日志形式广播给集群中的节点, 避免告警事件在集群中重复出现
  6. Notify Provider 模块用于提供通知的具体实现, 并会在本地记录日志

安装

AlertManager 提供了预编译的二进制文件 releases, 也支持使用容器的方式部署 prom/alertmanager, 详见 install

配置

AlertManager 通过 YAML 文件来配置, 命令行可以配置不可变的系统参数, 而配置文件定义的限制规则用于通知路由和接收者, 详见 configuration

认证

AlertManager 支持 basic 认证和 TLS, 通过 --web.config.file 指定配置文件, 详见 authentication

密码参考 bcrypt

启动

通过运行 alertmanager 执行文件启动服务

服务启动之后, 可以在控制台 /status 查看服务状态和配置信息

AlertManager 提供了一组管理 API, 以促进自动化和集成, 详见 management-api

  • GET /-/healthy 该端点用于显示服务的健康状态
  • GET /-/ready 当服务准备好时, 该端点返回 200 状态码
  • POST /-/reload 该端点用于触发重新加载服务的配置和规则文件

通知

根据 Prometheus 和 AlertManager 的告警配置, 会影响告警事件的通知逻辑和时间

  1. Prometheus 以 scrape_interval 采样周期的间隔从客户端拉取时序, 根据时序的采样值, 以 evaluation_interval 评估周期的间隔对告警规则的表达式 expr 进行评估;
  2. 当采集对象出现问题时, Prometheus 会持续尝试拉取时序, 直到经过 scrape_timeout 超时时间后停止;
  3. 告警规则的表达式评估为真时, 告警状态切换到挂起 (Pending);
  4. 告警状态挂起的持续时间超过 for 指定的时间后, 告警状态切换为触发(Firing), 并将告警事件发送给 AlertManager;
  5. 下个评估周期若表达式评估仍为真, 且持续时间超过 for 指定的时间, 则持续发送告警事件给 AlertManager;
  6. 直到表达式评估为假, 告警状态切换为非活动(Inactive), 并发送通知 AlertManager 告警事件已解决;
  7. AlertManager 接收到告警事件后, 先根据分组中的 group_wait 等待时间, 等待过后再发出告警通知;
  8. 同个分组的告警事件, 在等待时间内可能产生新的告警事件, 在前一个告警成功发出后, 经过 group_interval 周期时间后, 再发送新的告警通知;
  9. 如果分组里的告警一直没发生变化, 则在成功发出告警通知之后, 再经过 repeat_interval 重复周期的时间后, 重复发出相同的告警通知;
  10. 如果告警通知没有成功发出, 则等待 group_interval 周期时间后, 会再次尝试发出告警通知

分组

分组是指将性质类似的告警分成一个通知类, 不同分组之间互相独立, 互不影响, 分组可以聚合同类告警以减少告警数量, 通过精简的方式帮助分析问题, 详见 grouping

分组由配置文件中的路由树节点配置, 可按匹配规则细分成多个子节点, 子节点未配置的可选参数将从父节点继承配置

RoutingTreeEditor 是一个在线支持可视化的路由树编辑器, 通过生成直观的图例, 确保开发人员编写正确的路由分组

抑制

抑制是指当源告警通知已经发出时, 停止再发送目标告警的通知, 抑制主要用于消除冗余的告警, 详见 inhibition

静默

静默通过配置静默表达式, 使匹配的告警事件不再发送告警通知, 管理员可以在控制台 /silences 中添加静默表达式, 通常用于主机维护时或夜间运行批处理作业等场景, 添加临时的静默规则可以防止产生不必要的告警, 详见 silences

静默提供了一种简单的机制, 可以在给定的时间内忽略可存在的告警, 静默基于 matchers 配置实现, 类似于路由树, 对接收的告警事件进行检查, 判断它们是否和活跃的静默表达式匹配, 如果匹配成功, 就不会将这些告警发送给接收端

AMTool

AMTool 是用于与 AlertManager 架构中的 API 模块进行交互的命令行工具, 它可以使用 AlertManager 提供的查询语法, 进行告警事件的查询, 也支持管理服务的静默表达式

集群

为了避免单节点问题, 通常会对多个 AlertManager 进行集群部署, AlertManager 之间通过 Gossip 机制进行集群成员管理和成员故障检测, 以确保在多个 AlertManager 分别接收到相同告警事件的情况下, 只有一个告警通知被发送给接收端, 要创建 AlertManager 的高可用集群, 可使用 --cluster.* 标志进行配置, 详见 high-availability
